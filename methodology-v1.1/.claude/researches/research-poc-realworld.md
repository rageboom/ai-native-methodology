## 토론 주제 1: RealWorld 시드 적합성

**📘 공식문서 리서처**: RealWorld는 Greg Murray가 만든 표준 spec. 의도가 명확함 — "어떤 프레임워크라도 같은 기능을 구현". 따라서 비즈니스 도메인은 의도적으로 단순. PoC 시드로는 **방법론 검증에 적합**하지만 **한국 엔터프라이즈 검증에는 부적합**.

**🏢 사례 리서처**: Spring PetClinic은 더 단순하고 도메인이 명확하지만 FE가 빈약 (서버 사이드 렌더링 위주). RealWorld가 풀스택 검증에 더 좋다. 다만 **외부 의존성 부재**가 5.D 검증의 한계.

**🎯 Senior FE Engineer**: 현실적으로 PoC 시드는 "검증해야 할 것을 모두 가진 레포"가 없음. 두 가지 접근:
1. RealWorld로 **80% 영역** 검증 (5.D 외 모두)
2. 5.D는 사내 진짜 레포 PoC(다음)에서 검증

→ 합의: RealWorld는 **첫 번째 빠른 검증**으로 충분. 한계 명시 후 진행.

## 토론 주제 2: 누가 PoC를 실제로 수행하는가

이 질문은 **PoC의 가치를 결정**한다.

**🎯 Senior FE Engineer**: 가장 큰 함정은 "Claude가 너무 잘해서 빈틈을 못 발견"하는 것. Claude가 명세에 없는 결정을 자동으로 하면 → **그게 빈틈인지 모름**. 사람이 직접 명세를 보고 따라가면서 "어, 여기 어떻게 해야 하지?"가 발생해야 진짜 빈틈 발견.

**📘 공식문서 리서처**: 학계의 LLM-assisted code analysis 연구도 같은 한계 지적. 사람-AI 합동(human-in-the-loop)이 단독보다 결과물 품질 높지만, **빈틈 발견은 사람이 더 뛰어남**.

**🏢 사례 리서처**: 토스/카카오 등 사내 도구화 사례에서도 "PoC 1차는 사람, 2차부터 자동화"가 일반적.

→ 합의: **합동 모드 (옵션 C)** 채택. 단, 역할 분담 명시:
- **Claude**: 명세를 따라가며 1차 추출 시도
- **윤주스**: Claude의 결정을 검토하면서 "명세에 이게 있나?" 점검 → 없으면 finding으로 기록

## 토론 주제 3: 풀스택 vs BE만

**📘 공식문서 리서처**: 7대 산출물 중 7번 UI/UX는 FE 영역. BE만 하면 1~6만 검증되고 7번은 미검증.

**🎯 Senior FE Engineer**: 4시간 시간 제약을 고려하면 **BE 우선이 합리적**. 단, FE는 다음 PoC 또는 별도 세션으로.

**🏢 사례 리서처**: 대부분 PoC는 **한 도메인 한 단면**으로 시작. 풀스택 한꺼번에는 욕심.

→ 합의: **BE 우선** (Phase 0~6 중 5-2만 빼고). 5-2는 별도 세션 또는 다음 PoC.

다만 — **윤주스님이 FE 리드**이므로 5-2를 빼는 게 아쉬울 수 있음. 결정 권한은 윤주스님께.

## 토론 주제 4: 한 번에 vs Phase 단위로 끊기

**📘 공식문서 리서처**: 학계 LLM 분석 도구 사례에서 **단계별 검증**이 한꺼번에 보다 발견 빈도 높음.

**🎯 Senior FE Engineer**: Phase 단위로 끊는 가장 큰 이유는 **승인 게이트가 진짜로 동작하는지 검증**. 한꺼번에 하면 게이트 1번만 통과 (마지막). Phase 단위로 끊으면 8번의 승인 → 게이트 자체도 검증.

**🏢 사례 리서처**: 사내 도입 시 "phase별 승인"은 이해관계자 검토 시간 확보 효과도 있음.

→ 합의: **Phase 단위로 끊기**. 각 Phase 종료 시 윤주스님 승인 후 다음 진행.

## 토론 주제 5: 외부 레포 직접 접근 (web_fetch)

**📘 공식문서 리서처**: web_fetch 도구는 GitHub 파일을 직접 가져올 수 있음. 단, 레포 전체 구조 파악은 어려움 (각 파일 별도 호출 필요).

**🎯 Senior FE Engineer**: 가상 시나리오로 진행하면 "실제 RealWorld와 다름" 위험. 진짜 검증을 위해 web_fetch 활용해야.

**🏢 사례 리서처**: GitHub 공개 레포는 정상 사용 사례. 단, 토큰 제약상 **선택적 fetch** 권장 (모든 파일 X, 핵심 파일만).

→ 합의: **web_fetch로 핵심 파일만**. 디렉토리 구조는 GitHub API (또는 web_search) 활용.

## 5명의 가상 검토자 의견 — 추가 검증

가상으로 5명의 검토자가 PoC plan을 봤다고 가정하고 우려 사항 시뮬레이션:

### 검토자 A — 시니어 BE
> "Phase 4의 4영역 통합 알고리즘이 명세에 없는데 PoC에서 어떻게 통합할 거야? 그게 첫 번째 빈틈."

→ 예상 발견. F-001로 즉시 기록 준비.

### 검토자 B — DBA
> "RealWorld는 운영 DB가 없는데 §2.3 한계 §정합성 검증은 어떻게 검증?"

→ 정합성 검증은 단일 출처라 N/A 처리. 사내 진짜 PoC에서 검증.

### 검토자 C — 기획자/도메인 전문가
> "비즈니스 규칙 추출 후 도메인 전문가 인터뷰는 누가? RealWorld는 가상 도메인."

→ Medium의 알려진 정책으로 가상 답변. 사내 PoC에서 진짜 인터뷰.

### 검토자 D — 신규 입사자
> "이 PoC 산출물을 보고 다른 사람이 동일하게 재현 가능한가?"

→ 명세를 그대로 따라간 흔적이 명확해야 함. **단계별 명세 참조 라인 명시** 권장.

### 검토자 E — TF Lead (윤주스 본인 가상)
> "4시간 안에 정말 끝나? 7대 산출물 다 만들면서 빈틈도 기록?"

→ 솔직히 어려움. **BE 우선 + 7대 중 일부 빈약 인정**으로 시간 맞춤.
