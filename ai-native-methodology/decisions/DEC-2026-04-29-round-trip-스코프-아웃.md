## 컨텍스트

본 방법론 v1.1.2 까지 round-trip (산출물 → 신규 시스템 자동 생성) 미검증.
DEC-round-trip-미검증-인지 가 3 옵션 (1: 시도 / 2: PoC #03 / 3: 형식화 결합) 분기 대기.

사용자 결단 (2026-04-29):
> "일단 신규 시스템 검증 까지는 안해도 됨. 비즈니스 로직이나, 스팩을 뽑아내려 했던거는 로직은 언어 스팩이나 환경을 타는게 아니니까. 도움이 될수 있을것 같아서 그런거야 일단은 여기서는 신규 시스템 관련된거는 검증 안할꺼야. 다만 신규 시스템 만드는데 도움이 될만한게 있을지만 고민하는거지... 만약 기존 시스템에 버그나 위험한 것들, 혹은 부조리가 있다면 이것도 한켠에 다 기록하면 좋을것 같기도해."

## 결정

**round-trip (산출물 → 신규 시스템) 검증을 본 방법론 scope 에서 영구 제외**.

본 방법론의 가치를 다음과 같이 재정의:

> **"기존 시스템(legacy code) → 형식 명세 + 위험 기록"** 한 방향 추출기.
> 신규 시스템 구축은 사람의 책임. 본 산출물은 **참고 입력 자료**로만 활용.

## 근거

### 1. 가치 명료화
- 신규 시스템 자동 생성 = 70~80% 한계 (UI/UX / 운영 결정 / 사람의 개선 의도 형식화 불가)
- 자동 생성 검증 부담 (언어/스택/패러다임 다양성) > 가치
- 추출기로 한정하면 7대 산출물의 자기-완결성만 보장하면 됨

### 2. 산출물 이식성 (사용자 직관 정합)
"비즈니스 로직 / 스펙은 언어/환경 안 탐" — 이식 가능 산출물 5종:

| 산출물 | 언어 무관 | 환경 무관 | 신규 시스템 활용도 |
|---|---|---|---|
| **rules.json** (BR) | ✅ 100% | ✅ 100% | ★★★ 그대로 입력 |
| **domain.json** (Aggregate / UC) | ✅ 100% | ✅ 100% | ★★★ 그대로 입력 |
| **openapi.yaml** | ✅ 100% | ✅ 100% | ★★★ 그대로 입력 |
| **schema.json + erd.mermaid** | ✅ 90% (RDB 내) | ✅ DB 종류만 매핑 | ★★★ 입력 + 타입 매핑 |
| **antipatterns.json** | ✅ 패턴 단위 | ✅ | ★★★ 회피 목록 |

→ 5 산출물 모두 신규 시스템 입력으로 99% 이식 가능.

### 3. 위험 기록 가치
"기존 시스템에 버그나 위험한 것들, 혹은 부조리가 있다면 이것도 한켠에 다 기록"
- 이미 antipatterns.json (36 AP) + findings/poc-findings.md (76 finding) + composite view (PoC #02 4건) 으로 구현
- 강화: DEC-2026-04-29-안티패턴-마이그레이션-가이드 (별도 결단)

### 4. F-074 단방향 round-trip 은 유지
- F-074 (BR-USER-FOLLOW-NO-SELF-001) = 자연어 → 형식화 → 코드 생성 단방향
