## 1. 결단 본격

### 핵심 paradigm

본 방법론 chain harness 안 산출물 분리 axis = **stage 마다 다른 axis** (단일 paradigm 적용 ❌):

| Stage | 산출물 분리 axis | 근거 |
|---|---|---|
| analysis | 본격 분리 (BE 7대 + FE 8 / 이미) | 기술 자산 (technology-anchored) — 산출물 자체가 BE 또는 FE 본질 |
| **discovery** | **cross-cut 단일** (discovery-spec 1 산출물) | UC = 사용자 의도 anchor — BE/FE 양쪽 자연 가로지름 |
| **spec** | **cross-cut 단일** (behavior-spec + acceptance-criteria 각 1 산출물) | BHV/AC = 사용자 시나리오 anchor — Gherkin given/when/then 자연 cross-cut |
| **plan** | **layer 분기 본격** (task-plan 1 산출물 + tasks[].layer 필드) | TASK = 개발 작업 단위 — BE/FE 본격 다름 |
| test | framework 별 분리 본격 (test-spec.{junit5,vitest,playwright,...}) | framework diversity 본질 |
| implement | stack 별 분리 본격 (impl-spec.{be,fe} + 본문 stack 분기) | stack diversity 본질 |

### Story cross-cut 의 자연 귀결

사용자 결단 #6 = "Story 안에 BE Sub-task 도 본격 있을 수 있음".

→ Story 가 cross-cut anchor (BE/FE/DB/E2E 가로지름). discovery + spec stage 의 산출물 (UC / BHV / AC) = 사용자 의도/시나리오 anchor 라 **cross-cut 단일 산출물** 본격 정합. plan + test + implement stage = layer/framework/stack 분기 본격.

## 2. 구체 예제 — "비밀번호 변경 성공 시나리오"

```
discovery-spec (cross-cut / 단일):
  UC-001 = "user 비밀번호 변경"
    actors: [user]
    preconditions: [인증됨]
    postconditions: [새 비번 저장됨]
    source_grounded_evidence:
      - { kind: legacy_code, path: src/auth/PasswordCtrl.java:42 }

behavior-spec (cross-cut / 단일):
  BHV-001 = "비밀번호 변경 행동"
    preconditions: [인증, 기존_비번_일치]
    postconditions: [새_비번_hash_저장, 알림_발송]
    invariants: [비번_≥12자, 영문숫자특수문자_포함]

acceptance-criteria (cross-cut / 단일):
  AC-001 = Given 인증 / When POST /password-change / Then 200 + 알림
  AC-002 = Given 잘못된 old / When POST / Then 401

task-plan (layer 분기 / 단일 + tasks[].layer):
  TASK-001 (layer=fe / module=user-settings / ac_refs=[AC-001,AC-002])
    title: "PasswordChangeForm 컴포넌트 + 에러 toast"
  TASK-002 (layer=be / module=auth / ac_refs=[AC-001])
    title: "POST /api/password-change endpoint"
    openapi_endpoint_ref: { path: /api/password-change, operationId: changePassword }
  TASK-003 (layer=be / module=auth / ac_refs=[AC-001])
    title: "argon2 service"
  TASK-004 (layer=e2e / module=user-settings / ac_refs=[AC-001,AC-002])
    title: "playwright spec"

test-spec.junit5 (BE framework):
  TC-001 (target=TASK-002 / openapi_contract_ref: dredd)
  TC-002 (target=TASK-003 / property-based: schemathesis)

test-spec.vitest (FE framework):
  TC-101 (target=TASK-001 / vitest + RTL)

test-spec.playwright (e2e framework):
  TC-201 (target=TASK-004)

impl-spec.be (Spring stack):
  IMPL-001 = src/auth/PasswordController.java (commit_hash=abc...)
  IMPL-002 = src/auth/Argon2Service.java

impl-spec.fe (React stack):
  IMPL-101 = src/features/settings/PasswordChangeForm.tsx
```

## 3. 영향 면적 (schema 변경)

### task-plan.schema.json (#2 + #5 + #7 + #8 통합)

```json
{
  "tasks": [{
    "id": "TASK-001",
    "ac_refs": ["AC-001", "AC-002"],            // 1~3 묶음 (기존)
    "behavior_ref": "BHV-001",                   // (기존)
    "layer": "fe",                               // ★ 신설 enum: be/fe/db/e2e/infra
    "module": "user-settings",                   // (기존)
    "story_ref": "Story-101",                    // ★ #5 신설 (Jira mapping)
    "epic_ref": "Epic-10",                       // ★ #5 신설 (Jira mapping / FE 화면)
    "op_task_ref": null,                         // ★ #7 신설 (Story sibling / nullable)
    "openapi_endpoint_ref": {                    // ★ #8 신설 (BE 본격 필수)
      "path": "/api/password-change",
      "operationId": "changePassword"
    },
    "component_ref": {                           // ★ #8 신설 (FE 본격 필수)
      "package": "src/features/settings",
      "name": "PasswordChangeForm"
    },
    "estimation_ai": "30min",                    // (기존)
    "estimation_human": "10min"                  // (기존)
  }]
}
```

### test-spec.schema.json (framework 분기 본격 + #8 contract 강제)

`test_cases[].framework` enum 본격 확장:
- 기존: jest, vitest, junit5, pytest, mocha
- 신설: **schemathesis, dredd, pact, playwright, playwright-visual, axe-core, percy**

## 4. 산출물 file naming convention

### discovery / spec (cross-cut 단일)
- `.aimd/<scope>/discovery/discovery-spec.{json,md}`
- `.aimd/<scope>/spec/behavior-spec.{json,md}`
- `.aimd/<scope>/spec/acceptance-criteria.{json,md}`

### plan (단일 + layer 필드)
- `.aimd/<scope>/plan/task-plan.{json,md}` (단일 / tasks[].layer 안 BE/FE 분기)

### test (framework 별 분리)
- `.aimd/<scope>/test/test-spec.junit5.{json,md}` (BE Spring)
- `.aimd/<scope>/test/test-spec.vitest.{json,md}` (FE)
- `.aimd/<scope>/test/test-spec.playwright.{json,md}` (e2e)
- `.aimd/<scope>/test/test-spec.schemathesis.{json,md}` (BE contract / 본격 시행 시)
- `.aimd/<scope>/test/test-spec.axe-core.{json,md}` (FE a11y)

### implement (stack 별 분리)
- `.aimd/<scope>/impl/impl-spec.be.{json,md}` (Spring / NestJS / FastAPI 등 본문 분기)
- `.aimd/<scope>/impl/impl-spec.fe.{json,md}` (React / Vue 등 본문 분기)

## 5. cross_links 정합 (traceability matrix)

DO-178C / IEC 62304 bidirectional traceability axis 본격:

```
UC-* (discovery-spec)
  ↓ forward                    ↑ backward
BHV-* (behavior-spec)
  ↓                            ↑
AC-* (acceptance-criteria)
  ↓                            ↑
TASK-* (task-plan / layer 분기)
  ↓                            ↑
TC-* (test-spec.{framework})
  ↓                            ↑
IMPL-* (impl-spec.{stack})
```

★ chain-coverage-validator 안 본격 cross_links forward+backward 의무 강제. 누락 시 release-readiness block.

## 6. ratchet (legacy carry 대응)
