## 2. 핵심 발견 — schema.sql = ORM derivative 확정

### 2.1 결정적 증거 — Hibernate `NamingHelper` MD5 hash 패턴

`schema.sql` 의 **9 FK + 3 UQ 제약명 100% Hibernate auto-generated 패턴**:

| schema.sql:95 | `fki9gqdjcgtclgypxp2krye61n7` | FK (user_follow → users follower)  |
| schema.sql:98 | `fk474orxbed564dk0mvggvt4xon` | FK (user_follow → users following) |

**알고리즘 (Hibernate 6.5 source — `org.hibernate.boot.model.naming.NamingHelper#generateHashedFkName`)**:

```
prefix("FK") + base35( MD5( "table" + tableName +
                            "references" + referencedTableName +
                            "column" + sortedColumnNames ) )
```

- 본 PoC schema.sql 의 FK 명: `fk` + 25자 base35 lowercase = NamingHelper 패턴 정확 일치
- UQ 명: `uk` + 25자 base35 lowercase = `generateHashedConstraintName` 변형 패턴

**출처**:

- Hibernate 6.5 source: https://github.com/hibernate/hibernate-orm/blob/6.5/hibernate-core/src/main/java/org/hibernate/boot/model/naming/NamingHelper.java
- Vlad Mihalcea: https://vladmihalcea.com/hibernate-naming-strategy/

### 2.2 비교 — 수동 작성 DDL vs auto-generated DDL

| 분류           | 본 PoC 발견     | 수동 작성 DDL 패턴              |
| -------------- | --------------- | ------------------------------- |
| FK 명          | `fk + 25자 hex` | `fk_article_author` (사람-읽는) |
| UQ 명          | `uk + 25자 hex` | `uq_users_email` (사람-읽는)    |
| 사람-읽는 명명 | **0건**         | 일반적                          |

→ **본 PoC schema.sql 은 100% Hibernate auto-export 캡쳐본 확정**. 운영 DDL 또는 수동 SoT 가 아님.

### 2.3 Spring Boot 3.x 동작 검증 (Document sub-agent)

- 출처: Spring Boot 3.3 reference - https://docs.spring.io/spring-boot/3.3/how-to/data-initialization.html
- 공식 인용: **"Best Practice: Use only one initialization mechanism: Either Hibernate's `spring.jpa.hibernate.ddl-auto` Or script-based initialization (`schema.sql`) Or migration tools"**
- 본 PoC = **schema.sql + ddl-auto=create-drop 동시 사용 = 공식 anti-pattern 위반**
- 실제 동작: Hibernate `create-drop` 가 startup 마다 schema 재생성 → schema.sql 의 흔적은 사라지고 Hibernate 자체 산출물만 남음. schema.sql = documentation 목적의 reference 역할

## 3. 통합 우선순위 정책 (DEC-PHASE2-POC02-001)

### 3.1 v1.1.2 §3.4 Decision Tree 적용

```
Q1. 마이그레이션 도구 (Flyway/Liquibase) 도입 가능?
  → No (도구 미사용)
  ↓
  Q2. 운영 DB 존재?
    → No (H2 in-memory + create-drop)
    ↓
    공식 적용: ddl-auto=create-drop (테스트 한정)
              우선순위: DDL 파일 > ORM > ERD
```

### 3.2 본 PoC 의 실질 적용 — Decision Tree 무력화

**현행 v1.1.2 명세 한계** (DEC-PHASE2-POC02-003 / PF-PHASE2-META-001):

- Q3 (DDL 파일이 ORM auto-generated 인지) 분별 절차 부재
- 본 PoC schema.sql 이 ORM derivative 이므로 "DDL > ORM" 우선순위 무의미
- → **JPA Entity = SoT 단일** 채택

### 3.3 v1.1.2 명세 한계 권고 (PF-PHASE2-META-001 high promoted)

**권고 — phase-2-db.md §3.4 Q3 추가**:

```
Q3. DDL 파일이 ORM auto-generated 인가?
  ├─ Yes → DDL = ORM 동일 출처. 우선순위는 ORM 단일.

  └─ No  → 기존 우선순위 (DDL > ORM)
검증 단서:
- FK/UQ 제약명이 hash 패턴 (fk + 25자 hex / uk + 25자 hex)

```
