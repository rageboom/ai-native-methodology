## 2. ★ 핵심 발견 — TypeORM `synchronize:true` derivative

### 2.1 schema.sql 출처 분별 (★ G6 v1.2.2 sub-section (b) TypeORM 첫 외부 사용)

#### 일반 원칙 (G6 §2.1 모든 ORM 공통)

본 PoC 는 ORM derivative 4 표지 모두 적용:
1. **명명 규칙 일관성** — entity 데코 명시적 (5 중 4 = `@Entity('xxx')` / Comment 1 = default)
2. **컬럼 순서** — entity field 선언 순서 (사람 의도 ❌)
3. **주석** — entity 에 비즈니스 의미 주석 부재 (default)
4. **migration 도구 부재** — DDL 직접 산출 = ORM 자동 생성 강한 시그널 ✅

#### TypeORM sub-section (b) 적용

| 표지 | TypeORM default 패턴 | 본 PoC 일치 |
|---|---|---|
| naming | camelCase 보존 + entity 명 그대로 + relation FK = `[propertyName]Id` | ✅ `authorId` / `articleId` / `followerId` / `followingId` |
| 명시적 table 명 | `@Entity('xxx')` 우선 / 없으면 클래스 명 → 'comment' | ✅ 4 명시 + 1 default |
| FK 제약명 | `FK_{table}_{column}` 또는 자동 (해시 ❌) | ⚠️ TypeORM 0.2 default — 검증 의무 (Sprint 5) |
| UQ 명명 | unique:true 부재 = UQ 자체 부재 | ✅ **UQ 0건 확인** |
| 컬럼 순서 | field 선언 순서 | ✅ entity 와 일치 |

→ **schema.sql 은 별도 출처 자격 X**. ORM entity 메타가 ground truth. (G6 sub-section (b) 정합 / F-050 분별 절차의 TypeORM 변형 확정).

## 4. 정합성 검증 5종 (G6 §4 정합)

### 4.1 Type 정합 (ORM ↔ schema)
✅ 통과 — entity field 타입 = TypeORM 추정 DDL 타입 일치 (TypeScript 타입 시스템이 1차 검증). PoC #02 F-048 (TagJpaRepository 타입 오류) 같은 결함 **비재현** — TypeScript 정적 타입이 사전 차단 ✅. ★ 본 PoC 의 학습 효과 입증 신호 (Senior 위험 4 정합 — TypeScript 정적 검증 권위).

### 4.2 Cardinality 정합 (FK 다중성)
- ✅ Article-Comment OneToMany 명시
- ✅ User-Article OneToMany / ManyToOne 양방향 명시
- ✅ User-Article ManyToMany (favorites) JoinTable
- ❌ **Follows-User 관계 부재** (followerId/followingId 단순 number) — F-121 high
- ⚠️ Article.tagList vs TagEntity 관계 **명시 부재** — F-123 (의미 모순)

### 4.3 Nullable 정합
✅ entity @Column nullable 명시 / DDL 추정 NOT NULL 일관성 ★. 단 implicit FK column (authorId / articleId) 은 TypeORM default = nullable=true → ★ orphan article 가능 (사용자 삭제 시).

### 4.4 UQ / Index 정합
**❌ ★★ 비즈니스 BR (RealWorld 표준) 의 unique 요구가 DDL UQ 로 강제되지 않음**:
- BR-USER-USERNAME-UNIQUE: ❌ App pre-check 만 (UserService.create QueryBuilder)
- BR-USER-EMAIL-UNIQUE: ❌ 상동
- BR-ARTICLE-SLUG-UNIQUE: ❌ slugify random suffix 회피
- BR-FOLLOWS-PAIR-UNIQUE: ❌ App pre-check 만 (ProfileService.follow if (!_follows))

→ **race-safe 보장 0건**. 1중 race-prone (App). PoC #02 의 2중 (App pre + DB UQ) 보다 **더 나쁜 상태** ★★. F-120 critical 확정.

### 4.5 Length / Type 정합 (RFC / 표준 위반 후보)
- email VARCHAR(255) — RFC 5321 §4.5.3.1.1 max 254 — **준수 ✅** (PoC #02 의 30자 위반 비재현)
- article.body / comment.body VARCHAR(255) — TypeORM default. **장문 저장 의도면 text 권장 (F-133 신규 medium)**

## 7. 변경 권고 우선순위

### 7.1 즉시 수정 (critical) — F-120

```typescript
// src/user/user.entity.ts
@Column({unique: true})  // ★ 추가
username: string;

@Column({unique: true})  // ★ 추가
@IsEmail()
email: string;

// src/article/article.entity.ts
@Column({unique: true})  // ★ 추가
slug: string;

// src/profile/follows.entity.ts — 추가 + composite UQ
@Index(['followerId', 'followingId'], {unique: true})  // ★ 추가
```

+ race window 처리 (UserService.create / ProfileService.follow):
```typescript
try {
  return await this.userRepository.save(newUser);
} catch (err) {
  if (err.code === 'ER_DUP_ENTRY') {  // MySQL DUP
    throw new HttpException({errors: {...}}, HttpStatus.BAD_REQUEST);
  }
  throw err;
}
```

### 7.2 단기 (high) — F-121

```typescript
// src/profile/follows.entity.ts
@ManyToOne(() => UserEntity, {onDelete: 'CASCADE'})
@JoinColumn({name: 'followerId'})
follower: UserEntity;

@ManyToOne(() => UserEntity, {onDelete: 'CASCADE'})
@JoinColumn({name: 'followingId'})
following: UserEntity;
```

### 7.3 중기 (medium) — F-133 / F-135

```typescript
@Column({type: 'text'})  // varchar → text

// favoriteCount → 별 테이블 또는 atomic increment
```

### 7.4 장기 (low) — F-134
