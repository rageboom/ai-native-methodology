## 1. 컨텍스트

C-Sprint 1 (Phase 4.5 형식화 시범) 산출물 (FSM/Sequence/Decision Table/ADT) 검토 중 사용자 질문:

> "다이어그램이 틀릴수도 있나?"

이 질문은 본 방법론의 **다이어그램 자산이 절대값 100% 가 아니라 단계적 신뢰 자산** 임을 명시화하지 않았다는 갭을 노출했다.

기존 상태:
- ADR-003 (신뢰도 메타데이터) 는 산출물 *전체* 신뢰도를 다루나, **다이어그램 영역 별 단계 모델 없음**
- DEC-2026-04-29-다이어그램-신뢰-모델 + memory `feedback_diagram_trust_model.md` 만 존재 — 정식 ADR 미격상
- C-Sprint 1.5 의 시뮬 패널티, Sprint 4 의 자동 검증 도구 도입 후에도 정식 ADR 부재 → "이 다이어그램 신뢰도 = N%" 판단 근거 본체 부재

→ DEC-2026-04-30-M-묶음-갭-식별 P2 #5 로 등록. 본 ADR 로 격상.

### 1.1 6 가지 오류 원인 (사실 인정)

다이어그램이 틀릴 수 있는 6 메커니즘:
1. **추출 오류** — 사람·AI 한계 (코드 오독)
2. **모델 단순화 손실** — Mermaid 표현력 한계 (composite state nesting / parallel)
3. **Source of Truth 함정** — 코드 진실 가정 위험 (실제 prod 와 git 차이)
4. **형식 표기 자체 한계** — nested / parallel state 를 정확히 표현 못 함
5. **Drift** — 시간 경과 동기화 부재 (ADR-008 의 이중 렌더링 정합 보전)
6. **사람 모델 편향** — 추상화 임의적

본 방법론의 신뢰 모델은 위 6 메커니즘을 **각 단계마다 점진적으로 줄이는 정량 모델** 로 설계된다.

## 2. 결정

**Phase 4.5 형식 명세 + 모든 다이어그램 영역 (mermaid 산출물) 의 "단계적 신뢰 모델" 공식 채택**.

다이어그램은 "정답" 이 아니라 **"단계적으로 신뢰 쌓는 자산"**. 사용 시 신뢰도 명시 + 용도별 가이드 적용.

## 3. 결과 (Consequences)

### 3.1 좋은 점

- **다이어그램 자산의 솔직함** — "이 그림 어디까지 믿나" 답변 가능
- **AI persona 시뮬 → 진짜 도구로의 진화 경로 정량화** — Sprint 5+ 진짜 도구 도입 시 신뢰도 격상 명확
- **★★★ 본 방법론 차별점** — "AI persona vs 진짜 도구" 컬럼 분리. 외부 사례 부재 = 본 방법론 차별 자산 (테크기업 사례 research 시 직접 사례 fetch 실패).
- **자동 도구 단계 신설** (v1.2.1) — drift-validator / decision-table-validator 의 정량적 위치 명확

### 3.2 나쁜 점

- 모든 다이어그램 산출물에 메타 작성 의무 — 작성 비용 증가 (mitigated by `meta-confidence.template.yaml` v1.2.2)
- 단계 7 이상 (사람 / 형식 증명) 은 본 방법론 scope 외 → 신뢰도 95%+ 달성은 사용자 책임
- Phase 4.5 외 다이어그램 (architecture.mermaid / erd.mermaid / dependency-graph.mermaid) 은 본 모델 적용 후행 (v1.3+ 검토)

### 3.3 무시한 다른 옵션

- **단일 신뢰도 표기 유지** — 거부. 사용자 질문 ("틀릴 수도 있나") 의 본질 회피.
- **검증 단계 무한 세분화** — 거부. 7단계 + 8단계 (이론적) 가 PoC #02 누적 데이터 정합. 더 세분화는 작성 비용 폭발.
- **자동 도구 = 진짜 외부 도구로 통합 표기** — 거부. v1.2.1 자동 도구 (정규식 기반) 와 v1.3+ 진짜 외부 도구 (Semgrep 등) 는 신뢰도 기여 메커니즘 다름 (자동 도구 = 우리가 만든 알고리즘 / 진짜 도구 = 외부 입증된 분석기).
