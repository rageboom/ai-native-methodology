- 상태: 승인됨 (Accepted)

- 결정자: 윤주스 (TF Lead, Auto Mode 위임)
- 관련: ADR-001 (사상적 기반), ADR-003 (신뢰도 메타데이터), **ADR-008 (이중 렌더링 사상 — 짝)**, DEC-2026-04-29-다이어그램-신뢰-모델 (ADR 격상 모체), DEC-2026-04-29-static-tool-실행-의무화, DEC-2026-04-30-M-묶음-갭-식별, memory `feedback_diagram_trust_model.md`

## 1. 컨텍스트

C-Sprint 1 (Phase 4.5 형식화 시범) 산출물 (FSM/Sequence/Decision Table/ADT) 검토 중 사용자 질문:

> "다이어그램이 틀릴수도 있나?"

이 질문은 본 방법론의 **다이어그램 자산이 절대값 100% 가 아니라 단계적 신뢰 자산** 임을 명시화하지 않았다는 갭을 노출했다.

기존 상태:
- ADR-003 (신뢰도 메타데이터) 는 산출물 *전체* 신뢰도를 다루나, **다이어그램 영역 별 단계 모델 없음**
- DEC-2026-04-29-다이어그램-신뢰-모델 + memory `feedback_diagram_trust_model.md` 만 존재 — 정식 ADR 미격상
- C-Sprint 1.5 의 시뮬 패널티, Sprint 4 의 자동 검증 도구 도입 후에도 정식 ADR 부재 → "이 다이어그램 신뢰도 = N%" 판단 근거 본체 부재

→ DEC-2026-04-30-M-묶음-갭-식별 P2 #5 로 등록. 본 ADR 로 격상.

### 1.1 6 가지 오류 원인 (사실 인정)

다이어그램이 틀릴 수 있는 6 메커니즘:
1. **추출 오류** — 사람·AI 한계 (코드 오독)
2. **모델 단순화 손실** — Mermaid 표현력 한계 (composite state nesting / parallel)
3. **Source of Truth 함정** — 코드 진실 가정 위험 (실제 prod 와 git 차이)
4. **형식 표기 자체 한계** — nested / parallel state 를 정확히 표현 못 함
5. **Drift** — 시간 경과 동기화 부재 (ADR-008 의 이중 렌더링 정합 보전)
6. **사람 모델 편향** — 추상화 임의적

본 방법론의 신뢰 모델은 위 6 메커니즘을 **각 단계마다 점진적으로 줄이는 정량 모델** 로 설계된다.

## 2. 결정

**Phase 4.5 형식 명세 + 모든 다이어그램 영역 (mermaid 산출물) 의 "단계적 신뢰 모델" 공식 채택**.

다이어그램은 "정답" 이 아니라 **"단계적으로 신뢰 쌓는 자산"**. 사용 시 신뢰도 명시 + 용도별 가이드 적용.

### 2.1 신뢰도 정량 모델 (★ 7단계 + 도구 종류 구분)

| # | 검증 단계 | raw confidence | 누적 효과 | 도구 종류 | 도입 |
|---|---|---|---|---|---|
| 1 | 1차 작성 (사람·AI 단독) | **60~70%** | 베이스라인 | 사람 또는 AI 단독 추출 | v1.0 |
| 2 | + Cross-validation AI sub-agent (F-015 패턴) | **75~85%** | +5~15%p — 독립 sub-agent 비교 | AI sub-agent (Claude / Gemini 등) | v1.1 |
| 3 | + ★ v1.2.1 자동 검증 (drift-validator + decision-table-validator) | **78-85%** | +3~8%p — `.json ↔ .mermaid` 의미 동일성 자동 비교 + dmn-check 5종 | 자동 도구 (정규식 + 명시 알고리즘 — AI 추론 0%) | **v1.2.1** |
| 4 | + ~~Static Analyzer persona sub-agent~~ | 80~87% | ★ -5%p 패널티 (시뮬레이션) | AI sub-agent persona | (★ DEC-static-tool-실행-의무화 → 사용 시 simulation_reason 의무) |
| 5 | + 진짜 정적 도구 실행 | **85~92%** | +5~10%p — Daikon / Semgrep / PMD / SpotBugs / CodeQL 실제 실행 | ★ 진짜 외부 도구 (5종 물증 의무) | v1.2.1 (인프라) / 실 실행 v1.3+ |
| 6 | + Property test 실행 | **90~95%** | +5%p — jqwik / Vitest 실제 실행 | 실행 가능 명세 | v1.2.0 |
| 7 | + 사람 도메인 전문가 리뷰 | **95%+** | critical 영역 | 사람 | (선택) |
| 8 | + L4 (TLA+ / Coq) | **이론적 100%** | critical only | 형식 증명 | (★ scope 아웃) |

### 2.2 ★★★ 도구 종류 구분 의무 (본 방법론 차별점)

**AI persona vs 진짜 도구를 절대 동일 신뢰도로 표기 ❌** (DEC-2026-04-29-static-tool-실행-의무화).

| 도구 종류 | enum value | 신뢰도 기여 | 물증 의무 |
|---|---|---|---|
| AI sub-agent | `ai_subagent` | +5~15%p | sub-agent 결과 raw 보존 |
| ★ AI persona 시뮬 (Static Analyzer 흉내) | `ai_persona_simulation` | **-5%p 패널티** | `simulation_reason` 의무 |
| ★ 자동 검증 도구 (drift-validator / decision-table-validator) | `automated_tool` | +3~8%p | tool_version + 알고리즘 명시 |
| ★ 진짜 외부 도구 (Semgrep / PMD / Daikon / etc.) | `real_static_tool` | +5~10%p | **5종 물증 의무** (`tool_version` / `stdout_path` / `stderr_path` / `invocation_timestamp` + `duration_ms` / `result_hash` / `reproduction_command`) |
| Property test 실행 | `property_test` | +5%p | 실행 결과 보존 |
| 사람 도메인 전문가 | `human_review` | +5~10%p | 리뷰 기록 |
| 형식 증명 (TLA+ / Coq) | `formal_proof` | +5%p | 증명 stub |

→ `formal-spec.schema.json` `cross_validation.validators[]` 의 `real_tool: true` if/then 강제 (v1.2.1 적용).

### 2.3 사용 가이드 (모든 다이어그램 산출물에 명시 의무)

#### 신뢰해도 되는 용도

| 단계 ≥ | 용도 |
|---|---|
| 1 (60-70%) | 온보딩 자료, 코드 리뷰 보조 |
| 2 (75-85%) | PR 리뷰 시 drift 발견 |
| 3 (78-85%) | **이중 렌더링 정합 자동 검증 통과 산출물** 로서 신뢰 |
| 5 (85-92%) | AI 코드 생성 입력 (★ v1.2.0 묶음 L 가치명세) |
| 6 (90-95%) | 안티패턴 격상 근거 |
| 7 (95%+) | 사내 적용 시 critical 결정 근거 |

#### 신뢰하면 안 되는 용도 (모든 단계)

- critical 시스템 단독 근거 (단계 7+ 만 허용)
- 컴파일 없이 prod 배포 (실행 검증 의무)
- race-safety 단독 증명 (concurrency 형식 검증 별도 — Phase 4.5 invariants/property-test 만으로 부족)

## 3. 결과 (Consequences)

### 3.1 좋은 점

- **다이어그램 자산의 솔직함** — "이 그림 어디까지 믿나" 답변 가능
- **AI persona 시뮬 → 진짜 도구로의 진화 경로 정량화** — Sprint 5+ 진짜 도구 도입 시 신뢰도 격상 명확
- **★★★ 본 방법론 차별점** — "AI persona vs 진짜 도구" 컬럼 분리. 외부 사례 부재 = 본 방법론 차별 자산 (테크기업 사례 research 시 직접 사례 fetch 실패).
- **자동 도구 단계 신설** (v1.2.1) — drift-validator / decision-table-validator 의 정량적 위치 명확

### 3.2 나쁜 점

- 모든 다이어그램 산출물에 메타 작성 의무 — 작성 비용 증가 (mitigated by `meta-confidence.template.yml` v1.2.2)
- 단계 7 이상 (사람 / 형식 증명) 은 본 방법론 scope 외 → 신뢰도 95%+ 달성은 사용자 책임
- Phase 4.5 외 다이어그램 (architecture.mermaid / erd.mermaid / dependency-graph.mermaid) 은 본 모델 적용 후행 (v1.3+ 검토)

### 3.3 무시한 다른 옵션

- **단일 신뢰도 표기 유지** — 거부. 사용자 질문 ("틀릴 수도 있나") 의 본질 회피.
- **검증 단계 무한 세분화** — 거부. 7단계 + 8단계 (이론적) 가 PoC #02 누적 데이터 정합. 더 세분화는 작성 비용 폭발.
- **자동 도구 = 진짜 외부 도구로 통합 표기** — 거부. v1.2.1 자동 도구 (정규식 기반) 와 v1.3+ 진짜 외부 도구 (Semgrep 등) 는 신뢰도 기여 메커니즘 다름 (자동 도구 = 우리가 만든 알고리즘 / 진짜 도구 = 외부 입증된 분석기).

## 4. 적용 (Implementation)

### 4.1 산출물 메타 의무 (v1.2.2 — `meta-confidence.template.yml` 정합)

모든 mermaid / formal-spec 산출물의 `_manifest.yml` 또는 인접 메타에 `trust_level` 명시:

```yaml
trust_level:
  current: 0.78          # 0.60~1.00 (단계 §2.1 정합)
  current_step: 3        # 1~8 (§2.1 표 row #)
  validation_history:
    - {step: "1차 작성",       at: "2026-04-29", actor: "AI",         tool_type: "ai_subagent"}
    - {step: "cross-validation", at: "2026-04-29", actor: "Senior sub", tool_type: "ai_subagent"}
    - {step: "drift-validator",  at: "2026-04-30", actor: "automation", tool_type: "automated_tool"}
  next_validation: "real static tool execution (Sprint 5 carry-over)"
known_limitations:
  - "nested state 단순화 (메커니즘 #2)"
  - "race interleaving 정량 부재 (메커니즘 #4)"
```

### 4.2 v1.2.1 자동 도구 통합

- `tools/drift-validator/` 통과 시 단계 3 도달
- `tools/decision-table-validator/` 통과 시 단계 3 도달
- 둘 다 미실행 시 -5%p 패널티 (`v1.2.1 phase-4-5 워크플로우` 정합)

### 4.3 v1.3+ 진짜 도구 통합 (carry-over)

- Sprint 5 — `tools/static-runner/` 의 Semgrep / PMD 1회 실 실행 → 단계 5 도달
- 5종 물증 schema 검증 통과 시 신뢰도 85-92% 표기 허용
- 시뮬 시 -5%p 패널티 + `simulation_reason` 명시
