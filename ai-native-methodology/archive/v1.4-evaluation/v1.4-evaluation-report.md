### 6번 요구 — 인덱스 패턴 LLM 성능 영향 (답변)

Claude Code 의 컨텍스트 로딩 모델:
- `CLAUDE.md` 는 매 세션 자동 로드 (200 line 후 truncate).
- 거기 적힌 다른 파일 경로는 LLM 이 필요 시 Read tool 로 펼침 (lazy load).

영향:
- ✅ **짧은 인덱스 + 명확한 경로 hook** = 성능 ↑ (탐색 단축, 컨텍스트 절약).
- ⚠️ **너무 잘게 쪼개진 깊은 트리** (n>30 + depth>4) = 성능 ↓ (어디 봐야 할지 추론 비용).
- ⚠️ **인덱스 자체가 200 line 초과** = truncate 위험.

권고: 본 레포 CLAUDE.md (7.1K, 약 100 line) + STATUS.md / INDEX.md / methodology-spec/ 분할 패턴은 **현재 양호**. 추가 분할 시 "★ 표시 + 한 줄 hook" 유지 의무.

### ★★★ D.1 본 평가의 핵심 발견 — 정체성 충돌

사용자의 Q2/Q4 응답이 본 평가의 가장 중요한 인사이트를 드러냈다:

**충돌점**: 본 레포는 **"legacy 분석 도구"** 로 만들어졌고 정책도 그에 맞춰 작성되어 있다. 그런데 사용자 본래 목표는 **"AI-Native SDLC 전체 plugin"**. 두 정체성이 미스매치.

#### D.1.1 현재 정책의 위치와 한계

**위치**: `CLAUDE.md` 의 "★★★ 본 방법론 가치 명세" 섹션:

```
INPUT:  legacy 코드베이스
OUTPUT: 7대 산출물 + finding + antipatterns + migration-cautions
USE:    사람이 신규 시스템 구축 시 입력 자료 / 회피 가이드

"코드 → 형식 명세 + 위험 기록" 한 방향 추출기.
round-trip 검증은 영구 scope 제외.
```

**의미 분해**:
1. **INPUT = legacy code only** — 신규/수정/버그 use case 가정 X
2. **OUTPUT = 산출물 + 위험기록** — 운영(ops) 자산 X
3. **한 방향 (one-way)** — 산출물 → 코드 재생성 X
4. **round-trip 영구 제외** — 산출물이 코드와 동기화되는 검증 X

이 정책 하에서 본 레포는 **"코드 입력 → 명세 출력"** 의 단방향 추출기로 좁혀진다. 그래서 design/test/implement 가 placeholder 인 것도 자연스럽다 — 본 정책 안에서는 그 단계들이 **"본 plugin 의 일이 아니다"**.

#### D.1.2 사용자 멘탈 모델 vs 본 레포 모델

| 항목 | 사용자 멘탈 모델 | 본 레포 v1.5.0 |
|---|---|---|
| 입력 | 기획 / 디자인 / 비즈니스 요구 / (또는 legacy) | legacy code only |
| 첫 단계 | 분석 (요구사항 + 도메인) | analysis = legacy 추출 |
| 두번째 | 스펙 정의 | (없음 / placeholder) |
| 세번째 | 설계 (architecture + DB + API + UI) | design (placeholder) |
| 네번째 | 테스트 | test (placeholder) |
| 다섯번째 | 구현 | implement (placeholder) |
| 흐름 | 양방향 (산출물↔운영 sync 가능) | 한 방향 (round-trip 제외) |
| use case | legacy / 신규 / 수정 / 버그 | legacy 만 |

**두 모델의 정합 충돌**:
- 본 레포의 "analysis" = 사용자 멘탈 모델 첫 단계의 **legacy sub-mode 만**.
- 사용자 신규 use case (요구사항 분석) = 본 레포에 자리 없음.
- 본 레포의 "design" 자리가 사용자 멘탈 모델 어느 단계인지 모호 (스펙? 설계?).

#### D.1.3 그래서 본 평가의 진짜 결론

**본 레포의 정체성 결정이 모든 cleanup/추가/수정 결정의 prerequisite**. 정체성 결정 없이 cleanup 하면 잘못된 자산 제거 위험. 정체성 결정 없이 신규 자산 추가하면 정책 충돌.

**경로 옵션**:

| 경로 | 의미 | 영향 |
|---|---|---|

#### F.4.1 산출물 간 round-trip (★ 본 레포 현재 시행 중 ✅)

**예시**: state-map.json ↔ state-machine.mermaid (이중 포맷 정합).
- 분석 산출물 = JSON (LLM용) + Mermaid (사람용) 두 포맷.
- drift-validator 가 두 파일이 같은 의미인지 검증.
- 이건 "산출물 내부 정합" 이고 ADR-008 로 정책 정식화됨.

이건 **허용 + 시행 중**.

#### F.4.2 산출물 ↔ 코드 round-trip (★ 본 레포 영구 scope 제외 ❌)

**예시**:
- 정방향: 코드 → 산출물 (현재 분석 stage 가 함 — "추출")
- 역방향: 산출물 → 코드 (자동 생성)
- round-trip 검증: "역방향으로 만든 코드가 원본과 동치인가?" 또는 "코드 변경 시 산출물 자동 sync"

**본 레포 영구 제외 이유** (CLAUDE.md ★★★):
- AI 가 산출물 → 코드 자동 생성 = hallucination 위험 + 비즈니스 로직 누락 가능
- 본 레포는 산출물을 **"사람이 신규 시스템 구축 시 입력 자료"** 로만 정의
- 즉 사람이 산출물 보고 수동 구현 (또는 다른 도구 사용)

#### F.4.3 v2.0 결단 옵션 의미 재해석

| 옵션 | round-trip 의미 |
|---|---|
| (i) **round-trip 부분 허용** | implement stage 에서 산출물 → 코드 자동 생성 (보일러플레이트 / 스캐폴드 / CRUD 자동) **또는** 코드 변경 시 산출물 자동 sync. AI 가 SDLC 구현까지 적극 개입. |
| (ii) **영구 scope 보존** | 산출물 → 코드 자동 생성 ❌. plugin 은 명세만 제공. 구현은 사람 또는 다른 도구. plugin 의 구현 stage skill 은 "사람 가이드" 만 (예: 체크리스트, 모범 사례). |
| (iii) **하이브리드** | design 까지는 plugin 이 정식 채움 (wireframe / component-spec 자동 생성 가능). test/implement 는 사람 가이드만. |

#### F.4.4 사용자 처음 발화와의 정합

사용자 발화 4-2:
> 신규 기능은 기존의 문서를 참고 해서 **스팩정의, 설계, 테스트, 구현** 이 가능해야 한다.

★ "구현이 가능해야 한다" = **(i) 의 함의** (plugin 이 구현까지 도와야 함). 단 "AI 가 자동 생성" 인지 "AI 가 사람을 가이드" 인지 발화에서 명확하지 않음.

따라서 사용자 의도 더 구체화 필요:
- (i-strict) AI 가 코드를 자동 생성 (스캐폴드)
- (i-soft) AI 가 사람 코드 작성을 단계별 가이드 (현재 Claude Code 가 하는 일)
- (iii) 구현은 빼고 스펙·설계·테스트까지만

### F.5 사용자 응답 종합 — 본 평가의 잠정 결론

3차에 걸친 사용자 응답 패턴:
1. Q1 (cleanup) → "왜 묻는지 자세히" 요구 → 본인 확신 부족
2. Q2 (SDLC 단계) → "design 이 뭔가? 분석은 어디?" → 모델 명확화 요구
3. Q3 (use case) → ✅ "별도 entry flow" 즉답
4. Q4 (운영 sync) → "1번 정책 어디?" → 정책 위치 모름
5. Q1' (정체성) → "기획서 추출 가능?" → 기술 가능성 확인
6. Q1'' (정체성) → "round-trip 이 뭐야?" → 용어 명확화

**해석**: 사용자가 본 레포의 명세를 세부적으로 기억하지 못한 상태에서 평가 중. 다른 세션의 결정 (lifecycle-contract.md §가치 경계 deferral) 을 잊었거나 컨텍스트 부재. 단계적 명확화 후에야 결단 가능.

이는 사용자 메모리 (`feedback_session_handoff_drift.md`) 의 패턴과 일치 — "다른 세션 commit 인지 의무 / memory + STATUS + git log 동기화". 본 평가 세션 시작 시 이 동기화가 충분히 되지 않았을 가능성 (★ 단 본 평가가 "강제 재확인" 의 가치를 가짐).

**잠정 결론**: 사용자 결단을 한 번 더 묻되, **결단 보류 옵션** 도 정식 제공. 결단 보류해도 §G 최종 plan 은 "이번 turn 무위험 cleanup + v2.0 결단 후속 검토 항목 정리" 로 마무리 가능.

### F.6 4차 응답 (최종)
