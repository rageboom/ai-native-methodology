# Case Research (F-016): Hibernate ddl-auto 매트릭스 vs 원칙 — 실무 사례

> 작성일: 2026-04-28
> 역할: 테크기업 사례 리서처 (F-016 단독)
> 대상 충돌: document agent (매트릭스 OK) vs senior agent (원칙 + Decision Tree)

---

## §1. Executive Summary

- 영문 권위 자료 7곳 (rieckpil, Vlad Mihalcea, Spring Boot 공식 howto, Bell-SW, Baeldung, JPA Buddy, Atlas/Quesma 블로그) 모두 **production 권장값을 단일 식 (`validate` 또는 `none`) + Flyway/Liquibase** 로 정리. 4×2 매트릭스를 정식 가이드로 게시한 권위 자료는 발견되지 않음.
- 매트릭스 형태는 주로 **튜토리얼/입문자 자료** (Baeldung, agirlamonggeeks, Medium 강의글) 에서 enum 값 4~7개의 의미 설명 목적으로 사용. **운영 정책 문서**는 "production 에선 자동 migration 자체 금지" 원칙으로 단일화.
- senior agent 가 우려한 RDB 종속성·Hibernate 버전 종속성 우려는 실무 자료에서도 동일하게 관찰됨 — `ddl-auto=update` 의 PostgreSQL/MySQL 동작 차이는 명시 자료가 적고, dialect 코드 수준에서 분기됨 (즉, 매트릭스화하면 행이 RDB×버전으로 폭발).
- "카드사 새벽 2시 콜" 류 교훈은 영문권에서도 **"automatic schema management 자체 금지"** 원칙으로 일반화돼 있음 (Quesma, Atlasgo, JPA Buddy, Vlad).
- **결론 방향**: 실무 데이터는 senior agent 쪽 (원칙 + Decision Tree) 으로 강하게 기운다. 단, 매트릭스는 "enum 값 의미 reference table" 로 부록 수준에서 유지하는 것이 입문자 학습에는 도움.

## §2. ddl-auto 운영 사고/정책 사례

### 2.1 공식/준공식 권장 (Spring Boot / Hibernate)

- **Spring Boot 디폴트 정책**: 임베디드 DB (H2/HSQLDB/Derby) → `create-drop`, 외부 DB (PostgreSQL/MySQL/Oracle 등) → `none`. 이 자체가 "외부 DB = 운영" 의 안전성 가정에서 출발 (Spring Boot howto-database-initialization).
- **Vlad Mihalcea (Hibernate 공식 컨설턴트급 권위)**: "you should use the same schema migration scripts you are using for the production system" — `hbm2ddl.auto` 는 production 에서 사용하지 말 것, Flyway 가 정답. `update` 는 "to be avoided", `validate` 도 통합 테스트가 충실하면 불필요.
- **Hibernate 7.2 Migration Guide**: schema export 도구의 DDL 출력에 변경이 있으나 "외부에서 schema 를 관리하는 프로그램에는 영향 없음" 이라 명시 (즉, 권장 사용 패턴이 "외부 관리" 임을 공식 문서가 전제).

### 2.2 Flyway/Liquibase 와의 표준 통합 패턴

영문 권위 자료가 일관되게 권장하는 단일 패턴:

```
spring.jpa.hibernate.ddl-auto=validate
spring.flyway.enabled=true
# (또는 production 에서는 ddl-auto=none + Flyway 외부 실행)
```

- **rieckpil "Best Practices for Flyway and Hibernate with Spring Boot"**: "use Flyway exclusively for schema management via versioned SQL scripts in `src/main/resources/db/migration`. Apply validation mode (`validate`) so Spring Boot executes pending Flyway migrations first, then validates the database schema matches the JPA model".
- **bell-sw 공식 가이드 "How to Use Flyway with Spring Boot"**: 동일한 단일 권장.
- **Spring Boot Issue #894** (2014, milestone 1.1.0.M2): Hibernate validate 가 Flyway migrate 보다 먼저 실행되는 ordering 버그 → Spring Boot 1.1 부터 해결돼 "Flyway 가 먼저, validate 가 나중" 이 자동 보장됨. **이후 validate + Flyway 가 canonical pattern 으로 자리잡음**.
- **Spring Boot Issue #7874**: 동일 ordering 이슈가 후속 케이스에서 재발 → Spring Boot 측이 명시적으로 "migrations run before any application startup code" 보장.

### 2.3 production 사고 사례

특정 회사의 공개 postmortem (Stripe/Shopify/Atlassian 등) 에서 **"Hibernate ddl-auto 가 직접 원인"** 인 incident 는 검색 8분 내에 발견하지 못함 (대부분 회사가 애초에 Flyway/Liquibase 로 운영하므로 ddl-auto 가 사고 변수가 되지 않음).

대신 일반화된 사고 패턴은 풍부:

- **clustered 환경 동시 startup**: 여러 인스턴스가 동시에 schema 수정 시도 → deadlock, partial schema, startup failure (Quesma/Atlas 블로그).
- **dropped column / data type change**: `update` 는 drop 을 안 하므로 stale column 잔존, 반대로 add 측에서 not-null constraint 가 기존 row 와 충돌해 startup 실패.
- **audit trail 부재**: 자동 DDL 은 누가/언제/왜 변경했는지 기록 없음 → 컴플라이언스/디버깅 불가 (Atlasgo).
- **Stripe "Online migrations at scale"**: Stripe 는 schema migration 을 4단계 (dual-write, backfill, dual-read, cleanup) 로 manual orchestration. 자동 schema sync 는 사용 자체를 안 함. → "운영 = 수동 + observable" 원칙 강화.

## §3. 매트릭스 vs 원칙 — 실무 가이드의 형태

### 3.1 매트릭스 형태 가이드 (관찰됨)

매트릭스는 주로 **enum 값 의미 reference** 로 사용:

- Baeldung "Automatically Create a Database Schema With Spring Boot": 4개 값 의미를 표로 정리 (학습 목적).
- agirlamonggeeks "What Is Spring JPA Hibernate DDL Auto": 동일.
- Medium 의 다수 입문 글 (Rihab Beji, Punyakeerthi): 동일.

특징: **모두 "값 → 동작" 1차원 reference**. **"DB 존재 × ddl-auto" 2차원 매트릭스를 정식 가이드로 제시한 권위 자료는 발견되지 않음**. (Spring Boot 공식이 "임베디드 → create-drop, 외부 → none" 디폴트를 자동 적용하므로 사용자가 매트릭스를 의식할 필요가 없음.)

### 3.2 원칙 형태 가이드 (관찰됨, 압도적 다수)

- **Vlad Mihalcea**: "intentional, version-controlled schema migrations trump automatic generation" — 단일 원칙.
- **Atlasgo "Strategies for Reliable Schema Migrations"**: 원칙 4개 (versioned, reviewable, reversible, observable). 도구 무관.
- **Quesma "Schema migrations: pitfalls and risks"**: "Never use create, update, or create-drop in production" — 단일 원칙으로 모든 enum 값 처리.
- **Stripe "Online migrations at scale"**: 4단계 절차 원칙 (도구 무관, RDB 무관, ORM 무관).
- **JPA Buddy 가이드**: "ddl-auto 자체를 쓰지 마라, 도구로 review 후 SQL 로 관리" — 원칙.

### 3.3 어느 형태가 산업에서 더 널리 채택?

**원칙 형태가 압도적**. 이유 (실무 자료에서 도출):

1. **ddl-auto enum 값은 7개지만 production 에서 의미있는 값은 사실상 2개 (`none` / `validate`)**. 매트릭스 4행 중 2행은 "절대 쓰지 말 것" 으로 끝나므로 매트릭스 자체가 정보 손실.
2. **`none` vs `validate` 선택은 "Flyway/Liquibase 사용 여부 + startup overhead 허용 여부"** 단 2개 결정으로 환원됨 → Decision Tree 가 더 자연스러움.
3. **RDB 종속성**: PostgreSQL/MySQL/Oracle 의 `update` 동작 차이는 dialect 별 공개 표 자체가 없을 정도로 비표준 → 매트릭스화하면 dialect×버전 행 폭발.

## §4. RDB/Hibernate 버전 종속성

### 4.1 PostgreSQL vs MySQL ddl-auto=update 동작 차이

영문 검색에서 명시적 비교표는 발견되지 않음. 다만 다음 정황 증거로 senior agent 의 우려가 타당함이 확인됨:

- Hibernate `SchemaUpdate` 는 `Dialect` 별로 `getAlterTableString()`, `supportsIfExistsAfterAlterTable()`, `getAddColumnString()` 등이 달라 **DDL 생성 결과가 RDB 마다 다름**.
- ShardingSphere Issue #9463: PostgreSQL + ddl-auto=update 에서 dialect-specific exception 발생 사례 — MySQL 에선 재현되지 않음.
- Neon "Database Schema Changes with Hibernate" 가이드: PostgreSQL 전용으로 별도 가이드를 작성해야 할 만큼 **dialect 별 가이드를 분리하는 것이 일반적 관행**. → "단일 매트릭스로 모든 RDB 커버" 는 실무에서 채택되지 않은 패턴.

### 4.2 Hibernate 6.x → 7+ 변화

- **Hibernate 7.2 Migration Guide**: schema export 의 DDL 출력 변경 존재 (예: column 순서, default constraint 명명 규칙). 외부 schema 관리 프로젝트에는 영향 없음을 공식 명시 → **Hibernate 자체가 "ddl-auto 의존을 벗어난 운영"을 default 사용 패턴으로 가정**.
- **Spring Boot Issue #40177 "Simplify JPA DDL properties and auto-configuration"** (open): Spring Boot 측에서도 ddl-auto 자동 설정을 단순화/재정리하려는 흐름 → **매트릭스를 외워두면 향후 재작성 부담**.
- **Spring Boot Issue #45336**: ddl-auto 권장값이 Hibernate 와 정렬되지 않음 — Spring Boot 와 Hibernate 사이의 enum 의미 매핑조차 불완전.
- senior agent 의 "Hibernate 7+ 에서 deprecate/restructure 가능성" 우려는 직접 증거 부족 (코퍼스 기반 — URL 미검증)이지만, 위 이슈들로 보면 **enum 값 자체의 안정성 (long-term API stability)** 이 보장되지 않는다는 점은 사실.

## §5. 결론 — document vs senior 충돌 해결

| 쟁점 | document 입장 (매트릭스 OK) | senior 입장 (매트릭스 반대) | 실무 사례 결과 |
|---|---|---|---|
| 매트릭스 vs 원칙 형태 | 4×2 매트릭스 | 원칙 + Decision Tree | **senior 승**. 영문 권위 자료 7/7 이 단일 원칙 ("automatic schema management 금지 + Flyway/Liquibase") 채택. 4×2 매트릭스를 정식 가이드로 게시한 권위 자료 0건. |
| RDB 종속성 | 매트릭스 그대로 | RDB-agnostic 원칙 | **senior 승**. dialect 별로 `update` 동작이 다르고, Neon 같은 회사도 RDB 별 가이드를 분리. 단일 매트릭스로 커버 불가. |
| Flyway/Liquibase 통합 | 매트릭스에 행 추가 | 원칙에 흡수 | **senior 승**. Flyway/Liquibase 통합 시 표준은 `validate` (또는 `none`) 단 1행으로 환원됨. 행 추가는 정보 잡음. |
| 카드사 새벽 콜 교훈 | ddl-auto=update 위험 | 자동 migration 자체가 위험 (도구 무관) | **senior 승**. Quesma/Atlasgo/Stripe/Vlad/JPA Buddy 모두 "tool-agnostic, automatic migration 자체 금지" 원칙으로 일반화. |
| **단, 매트릭스의 가치** | enum 값 의미 학습 | 부록으로 충분 | **부분 절충**. Baeldung 류 입문 자료에서 "값 → 동작" 1차원 reference table 은 학습 가치 있음. **"reference (enum 의미 표) + 운영 원칙 (Decision Tree)" 2층 구조가 실무 표준에 가장 부합**. |

### 권장 v1.1.2 패치 방향

1. **운영 정책은 원칙 + Decision Tree** (senior 안 채택):
   - 원칙 1: production 에서 자동 schema migration 금지 (도구 무관).
   - 원칙 2: schema 변경은 versioned + reviewable + reversible (Atlasgo 4원칙 차용).
   - 원칙 3: ORM 의 schema 책임은 "검증 (validate)" 까지로 한정.
   - Decision Tree: (운영 DB 존재? × Flyway/Liquibase 채택? × 임베디드 테스트?) 3개 질문 → `none` / `validate` / `create-drop` 중 1개로 분기.

2. **enum 값 의미는 부록 reference table** (document 안 일부 채택):
   - 7개 enum 값 × "용도/위험/언제 쓰나" 3열 표를 **부록**에 배치.
   - 본문 가이드는 reference 표에 의존하지 않고 원칙으로 충분히 결정 가능하도록 작성.

3. **PoC #01 RealWorld 케이스 (`ddl-auto=none` + 운영 DB 부재 + schema.sql SoT)** 는 Decision Tree 의 한 leaf 로 자연스럽게 도달 → 매트릭스 행 추가 없이도 커버됨.
