# [YYYY.MM.DD] [기능명]
> Requirements Definition Document

> PM팀/개발팀이 직접 작성하는 통합 요구사항 정의서입니다. 대괄호 `[ ]`는 실제 작성 시 교체하고, 예시 행은 참고용이므로 지우거나 덮어씁니다. 기능 요구사항은 Jira 이슈 변환의 소스가 되므로, 변환 규칙은 별도 변환 스킬을 참조합니다.

| 항목 | 내용 |
|------|------|
| 상태 | 작성 |
| Type | [신규 / 개선 / 버그] |
| 버전 | v0.1 (초안) |
| 작성일 / 작성자 | [YYYY.MM.DD] / [PM 담당] |
| 요청자 / 요청부서 | [이름] / [부서명] |
| 요청 일시 | [YYYY.MM.DD] |

> 상태 값 안내
> - **작성** — 요구사항을 접수받아 정의서를 작성 중입니다. 01~04절을 작성한 후 검토 요청 시 상태를 **진행중**으로 변경합니다. 검토 없이 바로 확정되면 상태를 **완료**로 변경하고 기획/개발팀에 댓글을 남깁니다.
> - **진행중** — 검토 의견을 반영하여 기능 요구사항 및 예상 공수를 업데이트 중입니다. 05~06절(검토 내용·검토 결정)을 작성한 후 03~04절을 업데이트합니다. 업데이트 완료 후 확정되면 상태를 **완료**로 변경하고 기획/개발팀에 댓글을 남깁니다.
> - **완료** — 요구사항 정의가 확정되었습니다. 아래 댓글 양식을 기획/개발팀에 전달하세요.

## 3. 기능 요구사항 (Functional Requirements)

> PM이 Jira 티켓 구조를 미리 정의합니다. 기획팀·개발팀은 이 표를 기준으로 티켓을 생성합니다.
> 이슈유형 — `에픽` 메뉴 단위 / `스토리` 사용자에게 보이는 기능 / `작업` 내부 구조 변경 / `버그` 기존 동작 깨짐
> 계층 구조 — Initiative → Epic → Story · Task · Bug → Sub-task

| 번호 | 기능 | 이슈유형 | 상위 에픽 | 우선순위 | 설명 | 인수 조건 AC _(선택)_ | 비고 (요구사항 수집 이력) |
|------|------|---------|---------|---------|------|----------------------|------------------------|
| F-01 | [기능명] | 스토리 | [상위 에픽명] | Must | [기능 설명] | [검증 가능한 완료 기준] | RR-01 |
| F-02 | [기능명] | 작업 | F-01 | Should | [기능 설명] | [검증 가능한 완료 기준] | RR-01 |

> 우선순위: `Must` 필수 / `Should` 권장 / `Could` 선택
> 상위 에픽: 기존 에픽이 있으면 에픽명을, 하위 티켓이면 상위 번호(예: F-01)를 기재합니다.
