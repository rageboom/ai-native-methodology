# PM팀/개발팀 요구사항 정의서 작성 + Jira 변환 스킬

PM팀/개발팀이 요구사항 정의서를 표준 템플릿(`assets/요구사항_정의서_템플릿.md`)으로 작성하고, 완성된 문서를 Jira 이슈로 변환·등록하는 전체 워크플로우입니다.

전체 흐름은 4단계로 진행합니다.

```
1단계:   요구사항 정의서 작성 (상태: 작성 → 진행중 → 완료)
1.5단계: Confluence 위키 업로드
2단계:   Jira 이슈 변환 (검토용 목록 제시)
3단계:   승인 후 Jira 등록
```

---

## 1단계 — 요구사항 정의서 작성

### 시작 전 확인

1. `assets/요구사항_정의서_템플릿.md`를 view로 읽어 절 구조를 파악한다.
2. 사용자가 현업 요구사항·회의록·기존 문서를 제공했는지 확인한다. 없으면 기능명과 핵심 요구사항을 먼저 묻는다.
3. 템플릿의 대괄호(`[ ]`)와 예시 행을 실제 내용으로 교체한다. 정보가 없는 칸은 임의로 지어내지 말고 `[미확정]`으로 둔다.

### 상태 값 워크플로우

요구사항 정의서는 아래 흐름으로 상태를 관리한다. **상태가 바뀌는 시점마다 반드시 사람에게 확인을 받은 후 다음 단계로 넘어간다.**

**1. 작성**
요구사항을 접수받아 위키 페이지에 요구사항 정의서를 작성할 때 상태를 `작성`으로 표기한다.
이 단계에서는 아래 4개 절만 입력한다.
- 1절. 요구사항 수집 이력
- 2절. 배경 및 요청 내용
- 3절. 기능 요구사항
- 4절. 예상 공수

작성이 완료되면 사람에게 아래와 같이 확인을 요청한다.

> "1~4절 작성이 완료되었습니다. 문서를 검토해 주세요.
> 내용이 맞으면 '완료'로, 수정이 필요하면 '진행중'으로 알려주세요."

- **완료** 응답 → 상태를 `완료`로 변경하고 3단계로 이동한다.
- **진행중** 응답 → 상태를 `진행중`으로 변경하고 2단계로 이동한다.

---

**2. 진행중**
검토 의견을 반영하는 단계다. 상태를 `진행중`으로 표기하고 아래 두 절을 작성한다.
- 5절. 검토 내용
- 6절. 검토 결정

5~6절 작성이 완료되면 해당 내용을 기준으로 **3절(기능 요구사항)과 4절(예상 공수)를 업데이트**한다.

업데이트가 완료되면 사람에게 아래와 같이 확인을 요청한다.

> "검토 내용을 반영하여 3~4절을 업데이트했습니다. 문서를 다시 검토해 주세요.
> 내용이 맞으면 '완료'로 알려주세요."

- **완료** 응답 → 상태를 `완료`로 변경하고 3단계로 이동한다.

---

**3. 완료**
요구사항 정의가 확정된 상태다. 7절(산출물 및 관련 자료)을 채우고 기획/개발팀 위키 댓글에 아래 양식을 전달한다.

```
안녕하세요. [기능명] 요구사항 정의서가 확정되었습니다.
문서에서 Jira 티켓 위치 확인하시고, 기능 요구사항(3절) 표의 이슈유형·상위 에픽 기준으로 티켓을 생성해주세요.
```

### 표준 7절 구조

절을 임의로 추가·삭제하지 말고, 내용이 없으면 표를 비워두되 절 자체는 유지한다.

| 절 | 제목 | 상태 | 핵심 |
|----|------|------|------|
| 1 | 요구사항 수집 이력 | 작성 | 현업 원(raw) 요구사항, 출처, 반영 기능(F-xx) 연결 |
| 2 | 배경 및 요청 내용 | 작성 | 현행 문제점(선택), 요청 목적(선택), 요청 내용(필수) |
| 3 | 기능 요구사항 | 작성 | 기능(F-xx), 이슈유형, 상위 에픽, 우선순위, 설명, AC(선택) |
| 4 | 예상 공수 | 작성 | 단계별 공수(M/D·M/M), 담당부서/담당자, 전제 조건 |
| 5 | 검토 내용 | 진행중 | 건의 사항, 상세 내용, 검토 결과 |
| 6 | 검토 결정 | 진행중 | 검토자, 승인자, 결정 사항, 진행 여부, 최종 회신 |
| 7 | 산출물 및 관련 자료 | 완료 | Jira 티켓 목록, To-Do, 관련 자료 |

### 절별 주의사항

**1절** — 현업이 실제로 한 말을 원문에 가깝게 적는다. 정제는 3절에서 한다. 반영 기능(F-xx)으로 추적성을 연결한다.

**2절** — 요청 내용은 필수, 현행 문제점과 요청 목적은 해당하는 경우에만 작성한다.

**3절** — 가장 중요한 절이다. Jira 변환 소스가 되므로:
- 이슈유형을 반드시 채운다. 분류 기준은 아래 이슈유형 분류 기준 참조.
- 상위 에픽: 기존 에픽이 있으면 에픽명을, 하위 티켓이면 상위 F-xx 번호를 기재한다.
- 우선순위: `Must`(필수) / `Should`(권장) / `Could`(선택)
- 인수 조건(AC)은 선택이지만, "무엇을 만족하면 완료인가"를 검증 가능하게 적는 것을 권장한다.

**4절** — 공수는 M/D 또는 M/M으로 표기한다. 전제 조건은 타 팀·시스템이 먼저 제공해줘야 진행 가능한 사항이며, 없으면 생략한다.

**5~6절** — 상태가 **진행중**일 때만 작성한다. 6절 검토 결정 내용을 바탕으로 3~4절을 반드시 업데이트한다.

**7절** — 상태가 **완료**일 때 채운다. Jira 티켓 번호, To-Do, 관련 자료를 기록한다.

### 출력

완성된 요구사항 정의서는 `/mnt/user-data/outputs/요구사항_정의서_[기능명].md`로 저장하고 present_files로 전달한다.
전달 후 사람에게 아래와 같이 확인을 요청한다.

> "1~4절 작성이 완료되었습니다. 문서를 검토해 주세요.
> 내용이 맞으면 '완료'로, 수정이 필요하면 '진행중'으로 알려주세요."

## 1.5단계 — Confluence 위키 업로드

상태가 **완료**로 확정되면 요구사항 정의서를 Confluence 위키 페이지로 업로드한다. **사람의 확인을 받은 후에만 업로드한다.**

### 업로드 전 확인

사용자에게 아래 두 가지를 확인한다. 이미 알려준 경우 다시 묻지 않는다.

1. **스페이스 키** — 예: `HR`, `SNIT`, `EAM`. 모르면 `tool_search("confluence spaces")`로 목록을 조회해 선택하게 한다.
2. **상위 페이지** — 예: `"근태관리 > 리프레시 휴가"`. 모르면 해당 스페이스의 페이지 목록을 조회해 선택하게 한다.

### 페이지 제목 네이밍 규칙

```
[YYYY.MM.DD] [기능명] 요구사항 정의서
예) 2026.06.19 리프레시 휴가 결재 플로우 변경 요구사항 정의서
```

### 위키 업로드 절차

1. `tool_search("confluence page create")`로 Atlassian MCP의 페이지 생성 도구를 로드한다.
2. 아래 파라미터로 페이지를 생성한다.

```
- title:      [YYYY.MM.DD] [기능명] 요구사항 정의서
- spaceKey:   사용자에게 확인한 스페이스 키
- parentId:   사용자에게 확인한 상위 페이지 ID (또는 제목)
- content:    완성된 요구사항 정의서 전체 내용
```

3. 페이지 생성 완료 후 URL을 사용자에게 전달한다.
4. 생성된 URL을 7절(산출물 및 관련 자료)의 관련 자료에 기록한다.

### 업로드 후

페이지 URL을 전달한 뒤 사람에게 묻는다.

> "위키 업로드가 완료되었습니다. [페이지 URL]
> Jira 티켓 변환으로 넘어갈까요?"

## 2단계 — Jira 이슈 변환 (검토용 목록 제시)

3절 기능 요구사항을 읽어 Jira 이슈 후보 목록을 만들고, 사용자가 검토·승인한 뒤에 3단계(실제 등록)로 넘어간다. **승인 없이 먼저 등록하지 않는다.**

### 이슈유형 분류 기준

| 유형 | 이렇게 분류한다 | 예시 |
|------|----------------|------|
| 에픽(Epic) | 메뉴 하나 = 에픽 하나. 여러 기능을 묶는 상위 단위 | 내근무현황 메뉴, 휴가 신청 메뉴 |
| 스토리(Story) | 사용자 눈에 보이는 기능이 생긴다 | 연차 잔여일수 화면 표시, 엑셀 내려받기 |
| 작업(Task) | 사용자 화면은 그대로, 내부 구조만 바뀐다 | 배치 로직 리팩토링, SSO 연동, 인프라 설정 |
| 버그(Bug) | 원래 되던 동작이 깨짐 (신규 문서에는 보통 없음) | 집계 누락, 버튼 미동작 |

계층 구조: **Initiative → Epic → Story · Task · Bug → Sub-task**

### 변환 절차

1. 3절의 F-xx 목록을 순서대로 읽는다.
2. 각 기능을 이슈유형 분류 기준에 따라 판단한다. 3절의 이슈유형을 1차 기준으로 삼고, 틀렸으면 사유와 함께 수정한다.
3. 에픽이 될 묶음을 먼저 식별한다 (메뉴/화면 단위).
4. 전제 조건(4.2절) 항목은 작업(Task)으로 분류한다.
5. 검토용 목록을 다음 형식으로 제시한다:

```
[에픽] 시스템명 1차 릴리즈  ← 상위 묶음
  [스토리] F-01 기능명 — 인수조건 요약
  [스토리] F-02 기능명 — 인수조건 요약
  [작업]   P-01 전제조건 항목명
```

6. "이 목록으로 Jira에 등록할까요? 수정할 항목이 있으면 말씀해 주세요."라고 묻는다.

## 3단계 — Jira 등록

사용자가 검토용 목록을 승인한 뒤에만 실행한다.

### 이슈유형별 네이밍 규칙 및 Description 템플릿

#### 에픽 (Epic)

- **네이밍**: 실제 메뉴명 그대로 (예: 내근무현황, 휴가신청)
- **Description**:

```
■ 화면 개요
· 신규 — [메뉴명] 화면을 신규 제공한다.
· 기존 — 기존 [메뉴명] 화면에 [변경 내용] 추가
■ 화면 경로
/[메뉴 URL]
■ 비고
· 본 Epic은 서비스되는 동안 닫지 않으며, 이후 일감은 하위에 스토리/작업/버그로 계속 추가한다.
```

#### 스토리 (Story)

- **네이밍**: 사용자가 할 수 있는 것을 명사형으로 (예: 연차 잔여일수를 화면에 표시)
- **Description**:

```
■ 기능 목적
나는 [사용자]로서 [기능]을 하길 원한다. 왜냐하면 [이유] 때문이다.
■ 상세 내용
[기능 상세 설명]
■ 완료 조건
· [완료 기준 1]
· [완료 기준 2]
```

#### 작업 (Task)

- **네이밍**: 무엇을 어떻게 한다 (예: 근태 계산 배치 로직 리팩토링)
- **Description**:

```
■ 작업 내용
[작업 상세]
■ 작업 사유 (왜 Task인가)
화면은 그대로이고 내부 구조만 개선하는 작업으로 사용자 가치 변화가 없으므로 Task로 등록한다.
■ 완료 조건
· [완료 기준 1]
```

#### 버그 (Bug)

- **네이밍**: 어디서(발생위치) + 무엇이 + 어떻게 됨 (예: 내근무현황 1주차 근태 집계 누락)
- **Description**:

```
■ 전제조건
[재현을 위한 사전 조건]
■ 재현절차
[1, 2, 3 단계]
■ 재현결과
[실제로 나타난 잘못된 동작]
■ 기대결과
[원래 기대되는 정상 동작]
■ 문제
[무엇이 문제인지]
■ 해결 방향
[해결 접근]
```

### 등록 순서

계층 순서대로 등록해야 parent_key 연결이 가능하다.

1. **Initiative** — 시스템 단위. 이미 있으면 기존 키를 사용한다. 없으면 생성.
2. **Epic** — `jira_create`로 생성, Initiative 키를 parent_key로.
3. **Story·Task** — `jira_create`로 생성, Epic 키를 parent_key로.
4. **연결** — 필요 시 `jira_link`로 relates to / created by 연결.

### 등록 전 확인

- Jira 프로젝트 키를 사용자에게 확인한다 (예: HR, EAM). 모르면 `jira_projects`로 목록을 조회해 선택하게 한다.
- 이슈유형 한글 표기: `에픽` / `스토리` / `작업` / `버그` / `하위 작업`

### 등록 후

- 생성된 이슈 키(예: HR-10, HR-21)를 목록으로 정리해 보여준다.
- 등록 결과를 3절 기능 표의 비고란에 이슈 키로 기록하는 것을 권장한다.

## 자주 하는 실수

- 정보가 없는 칸을 그럴듯하게 지어내기 → `[미확정]`으로 두기.
- 3절 이슈유형을 비워두기 → 변환 소스가 불완전해진다.
- 공수를 기간(날짜)으로 적기 → M/D 또는 M/M 단위로 표기한다.
- 사람 확인 없이 위키에 바로 업로드하기 → 반드시 완료 확인 후 업로드한다.
- 스페이스 키·상위 페이지 확인 없이 업로드하기 → 잘못된 위치에 페이지가 생성된다.

- 검토 없이 Jira에 바로 등록하기 → 반드시 2단계 목록을 승인받은 뒤 등록한다.
- Epic보다 Story를 먼저 등록하기 → parent_key가 없어 계층이 깨진다. 반드시 상위부터.
