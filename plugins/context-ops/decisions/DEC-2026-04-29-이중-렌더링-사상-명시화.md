## 컨텍스트

Phase 4.5 형식화 도입 검토 중 사용자 명시화:

> "AI에게도 눈을 사람에게도 눈을 달아주고 싶은거야"

본 방법론의 모든 산출물 이 이미 이 사상으로 만들어져 있었으나, **사상 자체가 명시되지 않음**. ADR-001/002 의 진짜 의미가 이것임을 명시화 필요.

## 결정

**"이중 렌더링 사상" 을 본 방법론의 사상적 기반 (foundational principle) 으로 공식 명시화**.

정의:
- 모든 산출물은 **단일 진실 (Single Source of Truth)** 에서 출발
- **두 청중 (AI / 사람)** 을 위한 **두 렌더링** 동시 산출
- AI 눈: 기계 가독 (`.json`/`.yaml`, schema validation, AI 코드 생성 입력)
- 사람 눈: 시각/도식/요약 (`.mermaid`/`.md`, GitHub 자동 렌더, 리뷰·학습·수정)

## 근거

### 기존 방법론에 이미 구현됨 (사후 발견)
| 영역 | AI 눈 | 사람 눈 |
|---|---|---|
| Inventory | inventory.json | tree.md + stack-detection.md |
| DB | schema.json + schema.sql | erd.mermaid |
| Architecture | architecture.json | architecture.mermaid + architecture.md |
| Domain | domain.json | domain.mermaid + domain.md + use-cases.md |
| API | openapi.yaml | api.md |
| Rules | rules.json | rules.md |
| Antipatterns | antipatterns.json | avoid-list.md |

→ **새 사상 도입이 아니라 기존 무명 사상의 명시화**.

### 학문적 계보
- Donald Knuth, "Literate Programming" (1984) — 코드를 사람·컴퓨터 둘 다 읽도록
- 본 사상 = Literate Programming 의 AI 시대 재해석 (단일 파일 → 단일 진실 + 이중 렌더링)

### Phase 4.5 형식화 와의 관계
Phase 4.5 형식화 = 본 사상의 **미적용 영역 (서비스 행위·정책) 보완**:
| 신규 영역 | AI 눈 | 사람 눈 |
|---|---|---|
| Entity 생애주기 | state-machine.json (XState) | state-diagram.mermaid |
| UC 오케스트레이션 | sequence.json | sequence-diagram.mermaid |
| BR 정책 | decision-table.json | decision-table.md (표) |
| 타입 제약 | formal-spec.json (ADT) | domain.mermaid 보강 |
| Invariant | property.spec.ts (실행 가능) | invariants.md |

## 영향

### 즉시 적용
- 신규 산출물 추가 시 **이중 렌더링 의무화** (한쪽만 만들면 ADR-002 위반)
- 도구 선정 기준 명시:
  - 텍스트 기반 (git diff 가능)
  - GitHub 자동 렌더 (Mermaid 권고)
  - 외부 SaaS 의존 ❌
  - 락-인 위험 ❌

### Drift 방지 직결
- 두 렌더링이 어긋나면 = 단일 진실 손실
- CI 검증으로 일치 강제 (영역별 전략 A/B/C/D)

### ADR 격상 후보
본 결정은 **반복적·근본적 사상**이므로 ADR-008 또는 ADR-001 supersede 후보:
- 옵션 1: ADR-008 신규 ("이중 렌더링 사상")
- 옵션 2: ADR-001 supersede (Schema-First 사상에 이중 렌더링 명시 추가)
