## 1. 왜 이걸 만드는가? (목적)

### 1.1 만들고 싶은 것

레거시 코드를 받으면 AI 가 다음 7가지 문서를 자동으로 만들어주는 분석 환경:

| # | 문서 | 답하는 질문 |
|---|---|---|
| 1 | 아키텍처 | "이 시스템은 어떤 구조인가?" |
| 2 | 도메인 모델 | "어떤 개념과 규칙이 있는가?" |
| 3 | API 계약 | "어떤 요청/응답이 가능한가?" |
| 4 | DB 스키마 | "데이터는 어떻게 저장되어 있는가?" |
| 5 | 비즈니스 규칙 | "어떤 정책이 코드에 박혀 있는가?" |
| 6 | 회피해야 할 패턴 | "재구현 시 무엇을 피해야 하는가?" |
| 7 | UI/UX 명세 | "화면은 어떻게 구성되어 있는가?" |

### 1.2 이게 왜 필요한가?

레거시 분석은 사람 몇 명이 몇 달씩 매달려야 하는 작업. AI 에 맡길 수 있다면:
- **속도**: 1주일 안에 분석 완료
- **일관성**: 사람마다 결과 다른 문제 회피
- **재구현 입력**: 분석 결과를 그대로 AI 코드 생성의 입력으로 활용 가능

### 1.3 이번 PoC 가 검증할 3가지 기준

| 기준 | 무엇을 보는가 | 합격선 |
|---|---|---|
| **작동 여부** | 7가지 문서를 정말 자동 작성하는가 | 신뢰도 0.80 이상 |
| **방법론 빈틈 발굴** | 1회 시운전으로 명세의 부족한 부분을 발견하는가 | 5~15건 발견이면 건강함 |
| **실제 가치** | 분석한 코드의 진짜 문제를 찾아내는가 | 즉시 수정 필요한 결함 발견 |

## 2. 어떻게 실험했나 (실험 설계)

### 2.1 시운전 대상

이번에 사용한 코드: **RealWorld Conduit (Spring Boot 버전)**
- 공개된 학습용 블로그 서비스 코드 (회원가입/로그인/게시글/댓글/팔로우)
- 96개 Java 파일 / 약 4,700 줄 / 7개 테이블 / 22개 API

선정 이유: 사내 표준 스택 (Spring Boot + JPA) 과 유사하고, 1주일에 분석 가능한 규모. **이번 시운전 대상일 뿐, 진짜 평가 대상은 분석 환경 자체.**

### 2.2 분석 절차 — 6단계

```
1단계 (인벤토리: 어떤 파일/스택인지)
  → 2단계 (DB 스키마)
  → 3단계 (아키텍처)
  → 4단계 (도메인 + 비즈니스 규칙)
  → 5단계 (API)
  → 6단계 (회피 패턴 종합)
```

각 단계마다 **4가지 작업 원칙** 적용:
1. 작업 전 조사하고 계획서 작성
2. AI 보조 에이전트 3명을 띄워서 (공식 문서 담당 / 테크기업 사례 담당 / 시니어 엔지니어 역할) 토론하게 함
3. TF Lead 가 검토 후 코드 작성 승인
4. 실패하면 되돌리고 교훈 반영해서 1단계부터 다시

### 2.3 도중에 발견한 핵심 패턴

**문제**: AI 보조 에이전트가 학습 데이터에만 의존해서 50% 오차를 냄 (예: "Lombok 미사용" 이라고 답했는데 실제로는 사용 중).

**해결책 — 교차 확인 패턴**:
```
[메인 AI] 실제 코드 파일을 직접 읽음 (build.gradle, Entity 등)
   ↓
[보조 AI 3명] 같은 파일을 직접 읽도록 강제
   ↓
[메인 AI] 보조 AI 결과를 실제 코드와 비교
   ↓
50% 이상 차이 나면 즉시 "방법론 빈틈" 으로 등록
```

**결과**: 1단계 50% 오차 → 2~6단계 0% 오차. **이 패턴이 이번 PoC 의 가장 큰 자산**.

### 2.4 들어간 자원

| 항목 | 수치 |
|---|---|
| 분석 기간 | 1주일 |
| 메인 AI 의 파일 직접 읽기 | 누적 50회 이상 |
| 보조 AI 호출 | 18회 (6단계 × 3명) |
| 보조 AI 가 만든 자료 | 약 24,000줄 (조사 자료) |
| 보조 AI 실패 | 1회 (메인 AI 가 대신 작성해서 회복) |

## 3. 무엇을 얻었나 (결과)

### 3.1 기준 1 — 환경이 작동하는가? ✅ 합격

**7가지 문서 중 6개 자동 작성 + 평균 신뢰도 92%**:

| # | 문서 | 신뢰도 | 결과 |
|---|---|---|---|
| 1 | 아키텍처 | 0.91 | ✅ 작성 + Phase 1 추정과 Phase 3 확정의 차이까지 자동 기록 |
| 2 | 도메인 모델 | 0.90 | ✅ 핵심 개념/규칙 추출 |
| 3 | API 계약 | 0.93 | ✅ 표준 OpenAPI 형식 + 사내 메타 정보 분리 패턴 정립 |
| 4 | DB 스키마 | 0.92 | ✅ DB 와 코드 간 불일치 9건 자동 발견 |
| 5 | 비즈니스 규칙 | 0.85~0.92 | ✅ 13건 추출 (의도와 코드가 다른 규칙 1건 포함) |
| 6 | 회피 패턴 | **0.96** | ✅ 부분 작성 → 통합 패턴 작동 (15건) |
| 7 | UI/UX | — | 분석 대상이 백엔드 전용이라 해당 없음 |

→ **분석 환경이 실제로 작동한다는 1차 증명**.

### 3.2 기준 2 — 방법론 빈틈 발굴? ⚠️ 너무 많이 발견 (33건)

**합격선 5~15건 인데 33건 발견** — 두 가지 해석 가능:

| 해석 | 의미 |
|---|---|
| ✅ 좋은 신호 | "PoC 가 빈틈을 적극 발굴" — 방법론의 자기 개선 능력 작동 |
| ⚠️ 우려 신호 | "방법론 자체가 부실" — 기본 명세의 완성도 부족 |

**처분 결정**: PoC #2 (다른 스택) 한 번 더 돌려보고 합산해서 개선 결정. 이번 PoC 단독으로는 결정하지 않음 (1번 사례에 과적합 회피).

#### 방법론 빈틈 33건의 분류

| 분류 | 건수 | 의미 |
|---|---|---|
| 처리 완료 | 10 | 즉시 갱신해서 반영 (방법론 v1.1.2 패치) |
| 다음 버전에 반영 | 10 | PoC #2 와 합산 후 v1.2.0 격상 |
| 보류 | 13 | 더 많은 사례 필요 — 후속 PoC 후 재평가 |
| 거절 | 0 | 모두 정당한 빈틈 (잘못된 신고 0건) |

#### 즉시 갱신한 4건 (방법론 v1.1.2 패치)
- 인벤토리 데이터 규약 부재 → 신설
- 1단계 신뢰도 표 환경 의존성 미명시 → 보강
- 2단계 DB 통합 우선순위 분기 가이드 부재 → Decision Tree 추가
- 3단계 순환 의존성 분기 가이드 부재 → 의사결정 기록 (ADR-006) 신설

#### 다음 버전 (v1.2.0) 후보 7개 묶음
- A: 교차 확인 패턴 명세화 (이번 PoC 의 1순위 자산)
- B: 단계 간 추정 정정 절차
- C: 심각도(severity) 표준
- D: 데이터 규약 진화 (다중 카테고리 지원)
- E: 품질 추출 가이드
- F: 신뢰도 공식 보강
- G: API 메타 표준 가이드 — GitHub 의 `x-github` 패턴 모방

### 3.3 기준 3 — 실제 가치? ✅ 합격

**분석한 코드에서 즉시 수정 필요한 결함 4건 발견** — "이 환경이 실제로 가치를 만든다" 의 증명:

| 우선순위 | 결함 | 발견된 단계 | 의미 |
|---|---|---|---|
| 즉시 차단 | JWT 비밀키가 21바이트 (인터넷 표준은 32바이트 이상) + 코드에 하드코딩 | 4단계 + 6단계 | **보안 표준 7가지 위반** — 외부 출처 인용으로 정당화 |
| 즉시 차단 | 댓글 삭제 권한 조건문 논리 오류 (의도: A 또는 B / 실제 동작: A 그리고 B) | 4 → 5 → 6 단계 통합 | **단일 단계만으로는 못 본 결함** — 다단계 통합의 가치 |
| 1주 내 수정 | 이메일/사용자명 중복 가입 가능 (앱/DB/JPA 3중으로 unique 제약 부재) | 2단계 → 4단계 격상 | DB 불일치 보고서 → 회피 패턴 격상 절차 작동 |
| 1주 내 수정 | API 페이지 크기 제한 부재 (DoS 위험) | 5단계 보조 AI 권고 → 6단계 등록 | 보조 AI → 메인 통합 절차 작동 |

각 결함은 **방법론의 다른 자산이 작동했다는 증거**:
- 보안 결함 → 외부 표준 출처 인용 작동
- 논리 오류 → 다단계 통합 작동
- DB 결함 → 불일치 보고서 → 회피 패턴 격상 작동
- DoS → 보조 AI 권고 통합 작동

### 3.4 환경/도구 측면 작동 확인

| 자산 | 결과 |
|---|---|
| 4가지 작업 원칙 | 24단계 진행 / **되돌리기 0회** (사전 계획의 정밀도 입증) |
| 보조 AI 3명 패턴 | 18회 호출 / 17 성공 / 1 멈춤 (메인이 대신 작성해서 회복) |
| 교차 확인 패턴 | 1단계 50% 오차 → 2~6단계 0% 오차 |
| 빈틈 등록 시스템 | 등록 → 분류 → 다음 버전 격상 → 처분 4단계 모두 발현 (33건 처리) |
| 단계 간 추정 정정 | 1단계 추정 (Hexagonal 0.65) → 3단계 확정 (0.30) Δ -0.35 자동 기록 |

## 4. 무엇이 잘 안 되었나 (한계)

### 4.1 보조 AI 가 멈춘 사례 (6단계, 1회)

**증상**: 6단계의 시니어 엔지니어 역할 보조 AI 가 600초 동안 진행 안 됨.
**원인**: AI SDK 의 watchdog 한계 — 산출량이 많을 때 timeout 가능.
**대응**: 메인 AI 가 대신 약 700줄 작성. 결정의 정확도는 손상 없음.
**개선**: 다음 PoC 부터는 "보조 AI 멈춤 시 메인이 대신" 패턴을 명세에 정식 추가.

### 4.2 데이터 규약 (schema) 의 한계

**증상**: 회피 패턴 ID 형식이 단일 카테고리만 허용 (예: `AP-SECURITY-001` 만 가능, `AP-SECURITY-CONFIG-XXX` 불가).
**대응**: ID 변환 매핑 표를 산출물 메타에 명시.
**근본 해결**: v1.2.0 묶음 D 에서 일괄 처리 예정.

### 4.3 사내 권고 41건 통합 시 가독성

**증상**: 사내 적용 권고 41건을 하나의 체크리스트 파일에 통합하니 길어짐 (약 330줄).
**대응**: 심각도별 + 영역별 + 종합 점검 + 부록 매트릭스로 4단계 분류.
**검증 필요**: PoC #2 + 사내 적용 시 실제 가독성 평가.

### 4.4 보조 AI 들의 의견 충돌 해소가 1번 사례에 과적합 우려

**상황 (6단계)**:
- 사례 담당 AI: "여러 회사가 JWT/인증 패턴을 묶어서 다룬다 → 하나로 묶어서 등록 권고"
- 문서 담당 AI: "표준 분류는 개별 결함 단위 → 분리 등록 권고"
- 시니어 AI: 멈춤 → 메인이 대신 결정

**결정**: 분리 등록 채택 (1번 사례 과적합 회피 원칙).
**한계**: 이 결정의 정합성은 PoC #2 + 사내 적용 후 검증해야.

### 4.5 빈틈 33건 — 이번 PoC 만으로 결정 부족

§3.2 에서 다룬 양분 해석. 단일 PoC 데이터로는 v1.2.0 격상 결정 보류 — PoC #2 가 두 번째 데이터셋 역할.

### 4.6 5단계에서 4단계 산출의 분류 오류 3건 발견

**상황**: 5단계의 교차 확인 작업 중에 4단계 산출의 분류 오류 (한 사용자 조회 항목이 시스템 내부 로 잘못 분류 등) 가 뒤늦게 발견됨.
**한계**: 단계별 검증 게이트가 충분히 정밀하지 않음 → 다음 단계 진입 후 정정 필요.
**개선**: 다음 PoC 에서는 단계 진입 전 더 정밀한 검증 게이트 필요. v1.2.0 묶음 F (신뢰도 공식 보강) 와 연계 처리.
