#### DRIFT-002: user_followings 의 의미적 관계 (ManyToMany vs OneToMany)

```yaml
id: DRIFT-002
severity: medium
type: semantic_relationship_mismatch
table: user_followings
description: |
  DDL 측: 복합 PK (follower_id, followee_id) + 양쪽 FK→users.id CASCADE
         → 사실상 다대다 (User ↔ User self-reference)
  JPA 측: User.@OneToMany Set<User> followings + @JoinTable(name="user_followings",
         joinColumns=@JoinColumn(name="follower_id"), inverseJoinColumns=@JoinColumn(name="followee_id"))
         → 단방향 OneToMany 표현 (드문 패턴)
  
  의미적 차이:
  - DDL: 양방향 follow 관계 가능 (A→B, B→A 둘 다 존재 가능)
  - JPA OneToMany: 한 follower 가 여러 followee 를 가지는 단방향 관계
  
  실제 의도 (RealWorld spec): 단방향 follow — JPA 표현이 더 정확.
  단, DDL 표현 자체는 ManyToMany 와 동일 — 외부 데이터 마이그레이션 시 혼동 가능.

orm_evidence: "User.java:Set<User> followings + @OneToMany + @JoinTable"
ddl_evidence: "schema.sql user_followings 테이블 (복합 PK + 양쪽 FK)"
recommendation: |
  Phase 4 도메인 의도 검증 — Profile.follow() 메서드 확인.
  단방향 follow 가 의도면 명세 §3.4 의 ORM 우선 (Phase 2 명세 반영) — DDL 의 양방향 가능성은 제약 안 됨.
```

→ schema.json 의 컬럼 타입/제약은 DDL 그대로 채택. JPA 측 차이는 `drift_note` 로 보존.
