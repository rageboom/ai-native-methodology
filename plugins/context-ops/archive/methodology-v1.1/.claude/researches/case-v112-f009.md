# Case Research (F-009): 환경별 vs 결정성 — 신뢰도 분류축 실무 사례

> 작성일: 2026-04-28
> 역할: 테크기업 사례 리서처 (F-009 단독)
> 대상 충돌: Document agent (환경별 분리) vs Senior agent (결정성 단일축)

---

## §1. Executive Summary

산업계 8개 도구/시스템 (Sourcegraph, Semgrep, CodeQL, SonarCloud, GitHub Linguist, tree-sitter, Glean, Kythe) 조사 결과, **신뢰도 차이를 "환경(env)" 으로 노출하는 사례는 SonarCloud 의 shallow-clone WARNING 1건뿐** 이고, 나머지는 모두 **"분석 정확도 모드(precision tier)" 또는 "결정성(determinism)" 축으로 사용자에게 노출**한다. 특히 Sourcegraph 의 "precise vs search-based" 2-tier 와 Semgrep 의 "HIGH/MEDIUM/LOW confidence" 3-tier 는 **환경 enum 폭발 없이 단일 축으로 신뢰도 차이를 표현**하는 강력한 선례. CodeQL 의 `@precision` 메타데이터는 결정성 컬럼화의 정수. **결론: senior agent 의 "결정성 축 단일 표 + 컬럼화" frame 이 산업 표준에 가깝다.** 단, SonarCloud 처럼 **"환경 제약을 WARNING/note 로 부가 표기"** 하는 패턴은 채택 가치 있음 — 즉 "결정성 축 단일 표(주축) + 환경 caveat 주석(보조)" 하이브리드를 권장.

---

## §2. 코드 분석 도구의 신뢰도/정밀도 환경 종속

### 2.1 CodeQL `@precision` 메타데이터 — 결정성 축 컬럼화의 정수

GitHub CodeQL 은 모든 query 파일의 메타데이터에 `@precision` 키를 강제. 값 enum 은 `low / medium / high / very-high`. `@problem.severity` 와 함께 query 결과가 GitHub UI 에 표시되는 방식을 결정. (출처: codeql.github.com 메타데이터 문서)

핵심 시사점:
- **환경 (deep clone vs sparse) 과 무관하게** query 자체의 정밀도를 메타데이터로 선언.
- 신뢰도가 "환경" 이 아니라 **"분석 단위(query) 의 본질적 속성"** 으로 모델링됨.
- query suite 에서 `@precision very-high` 로 필터링 가능 (결정성 컬럼이 1급 시민).

→ F-009 적용: 각 산출물 항목 (LOC, Tree, ORM) 의 신뢰도를 **"환경별로 다시 계산" 하지 말고, 항목 자체의 본질적 정밀도 컬럼**으로 표현.

### 2.2 Sourcegraph "precise vs search-based" — 2-tier 단일 축

Sourcegraph 는 코드 navigation 신뢰도를 **2-tier 단일 축**으로 노출:

| Tier | 정확도 | 결정성 | 환경 요구 |
|---|---|---|---|
| **Precise** (LSIF/SCIP indexer) | compiler-accurate, 100% | deterministic (인덱스 결과) | indexer CI 셋업 필요 |
| **Search-based** (heuristic) | false positive/negative 발생 | stateless, 휴리스틱 | 즉시 사용 가능 |

핵심 인용: *"Sourcegraph automatically uses Precise Code Navigation whenever available, and Search-based Code Navigation is used as a fallback when precise navigation is not available."*

→ F-009 적용: **단일 신뢰도 축 + tier enum (deterministic / heuristic)** 으로 표현 가능. 환경 차이는 "precise indexer 가 동작했는가" 여부로 1차원 축에 흡수. **이는 senior agent 의 "결정성 축" frame 과 정확히 일치**.

### 2.3 SonarCloud shallow vs full clone — 환경별 분리가 아닌 "WARNING + note"

SonarCloud 는 shallow clone 환경에서:
- 분석 자체는 수행 (실패하지 않음).
- **WARNING: "Shallow clone detected during the analysis"** 를 분석 결과에 부가.
- 영향 범위 명시: SCM blame info 부정확 → issue auto-assignment 영향.
- 권장 조치: `fetch-depth: 0` 으로 disable shallow.

→ F-009 적용: **환경별 표를 분리하지 않고**, 단일 표 + "환경 제약 note/WARNING 컬럼" 으로 표현. document agent 의 "표 분리" 는 산업 사례에 없음. SonarCloud 패턴은 "결정성 축 단일 표 + 환경 caveat" 하이브리드의 실제 구현.

### 2.4 Semgrep snippet vs full project — 3-tier confidence

Semgrep 은 두 차원에서 신뢰도 표현:

(1) **분석 모드 (CE vs Pro)**:
- Community Edition: 단일 함수/파일 boundary 내 분석. true positive 44%.
- Pro/AppSec Platform: cross-file/cross-function/data-flow. true positive 75%, false positive -25%.

(2) **Rule-level confidence enum** (메타데이터):
- `HIGH`: metavariable-comparison/taint mode 사용, true positive 위주.
- `MEDIUM`: 동일 features + 일부 false positive 허용.
- `LOW`: dangerous look-alike, false positive 다수.

→ F-009 적용: Semgrep 도 **"환경" 이 아니라 "분석 모드(분석 단위 크기)" + "rule 본질적 confidence"** 의 2축으로 분리. 환경 enum 이 아닌 **"분석 깊이/범위" 축**으로 모델링. tier 수가 명확히 제한 (3개 고정) → enum 폭발 방지.

## §3. 분류축 — "환경" vs "결정성"

### 3.1 산업에서 "환경별" 분류 사용 사례

조사 결과 **순수 환경별 분리표 (env=A 표 / env=B 표 분리)** 사용 사례는 **0건**. 가장 가까운 사례:

- **SonarCloud**: shallow vs full 환경 차이를 인식하지만 **WARNING 부가** 형태 (표 분리 X).
- **Linguist**: byte 기반 통계는 **환경(local clone vs Languages API) 무관하게 동일** — Linguist 는 의도적으로 LOC 변환을 거부, byte-only 통계만 노출하여 환경 종속성을 회피.

→ **document agent 의 "환경별 표 분리" 권장은 산업 사례 부재**. 학술/blog 에서도 환경 enum 폭발 우려가 senior agent 와 동일하게 제기.

### 3.2 산업에서 "결정성" 분류 사용 사례

다수 채택:

| 시스템 | 결정성 축 표현 |
|---|---|
| Sourcegraph | precise (deterministic) / search-based (heuristic) |
| CodeQL | `@precision low / medium / high / very-high` 컬럼 |
| Semgrep | rule-level `HIGH / MEDIUM / LOW` confidence 컬럼 |
| tree-sitter | clean tree / ERROR node / MISSING node (parse 결과 결정성 노드 분류) |
| Glean (Meta) | facts 데이터베이스 — "AST-derived 는 deterministic, LLM-extracted 는 stochastic" 명시적 대비 (arxiv 2601.08773v1 인용) |

→ **결정성 축은 산업 표준**. 특히 arxiv 논문에서 *"DKB's graph construction is deterministic (AST-derived) given a fixed codebase and parser version, LLM-KB's extraction is inherently stochastic"* 직접 인용 — senior agent 의 frame 과 정확히 일치.

### 3.3 어느 frame 이 더 널리 채택?

**결정성(precision/confidence) 축 압승**:

- 8개 시스템 중 **결정성 축 컬럼화: 5건** (CodeQL, Sourcegraph, Semgrep, tree-sitter, Glean/Kythe 간접).
- **환경별 분리표: 0건**.
- **환경을 부가 caveat 으로 표기: 1건** (SonarCloud WARNING).
- **환경 종속성 의도적 회피: 1건** (Linguist byte-only).
- **명시적 메타데이터 없음: 1건** (Kythe — schema 만 정의, precision 메타데이터 부재).

산업의 frame 선택 logic 을 추론하면:

> 사용자에게 보여줄 신뢰도는 **"이 결과를 얼마나 신뢰할 수 있는가"** (정밀도) 이지, **"어느 환경에서 돌렸는가"** (구현 detail) 가 아니다. 환경은 정밀도를 결정하는 *입력* 일 뿐, *출력* 으로 노출하면 추상화 누수.

## §4. byte→LOC 변환 — 표준 부재 재확인

### 4.1 cloc / scc / tokei 의 LOC 측정 정확도 비교

세 도구 모두 **물리 LOC (physical lines)** 측정에 집중. logical/SLOC 변환은 도구마다 상이:

- **cloc**: 가장 오래된 표준, blank/comment/code 분리, 다수 언어 지원. 속도는 가장 느림 (tokei 의 1/20).
- **tokei**: Rust 구현, 병렬 파싱, 가장 빠름 (Linux 기준 +10ms 차이로 1위).
- **scc**: Go 구현, 복잡도(complexity) 계산 + COCOMO 추정 추가, 중복 파일 제거 지원. macOS/Windows 1위.

핵심: **세 도구 모두 "byte → LOC 변환 공식" 은 제공하지 않음**. 모두 파일 내용을 라인 단위로 직접 파싱. → byte/35 추정 같은 휴리스틱은 **학술/도구 표준 부재** (코퍼스 기반 — URL 미검증).

### 4.2 GitHub Linguist bytes-only 정책의 의도

Linguist 공식 문서:
- *"The percentages are calculated based on the bytes of code for each language as reported by the List Languages API."*
- 의도적 design decision: **단순성 + 파일 크기 분포 반영**.
- 알려진 한계: 큰 HTML 1개가 작은 Java 수십 개를 압도 가능 (counter-intuitive).

핵심 시사점: GitHub 도 **byte→LOC 변환을 의도적으로 거부**. *"LOC 환산은 학술 표준이 없으므로 byte 만 노출"* 이 GitHub 의 입장. → F-009 의 LOC 신뢰도 0.55 (byte/35 추정) 는 **본질적으로 부정확** 함을 산업이 이미 인정. 결정성 축에서 LOC 추정은 항상 "stochastic/heuristic" tier 로 분류되어야 함.

## §5. 결론 — document vs senior 충돌 해결

| 쟁점 | document (환경별 분리) | senior (결정성 축) | 실무 사례 |
|---|---|---|---|
| 분류축의 정당성 | 환경별 표 (git_clone / web_fetch / api_only) | 결정성 단일 표 + 컬럼 | **senior 우세**. 환경별 분리 0건, 결정성 축 5건. |
| enum 가지수 | 3개 (환경별) — senior 우려: 5개/7개 폭발 | 3개 (deterministic/snapshot/stateful) | 산업은 모두 **3-tier 고정** (CodeQL 4-tier 예외). enum 폭발 방지가 표준. |
| DRY 원칙 적용 | 표 분리 → 항목당 N회 중복 | 단일 표 + 컬럼 → DRY 유지 | **senior 우세**. SonarCloud 도 표 분리 X. |
| 환경 정보 표현 | 표 분리로 1급 시민 | 단일 표 + caveat note | **하이브리드**: SonarCloud "WARNING + note" 패턴. |
| 사용자 인지 부하 | 환경 결정 → 표 선택 → 항목 조회 (3-step) | 항목 조회 → confidence/precision 컬럼 직독 (1-step) | **senior 우세**. Sourcegraph 가 자동 fallback 하는 이유. |
| 추상화 누수 위험 | 구현 detail (환경) 노출 | 출력 정밀도만 노출 | **senior 우세**. 산업 표준은 출력 추상화. |

### 최종 권고 (사례 기반 종합)

**senior agent 의 "결정성 축 단일 표 + 컬럼화" frame 을 주축으로 채택**. 단, document agent 의 "환경 정보가 영향을 준다" 는 사실은 SonarCloud 패턴으로 보존:

```
결정성 축 단일 표 (주)
├─ 컬럼: 항목 / 신뢰도 (0.0~1.0) / 결정성 tier (deterministic/snapshot/heuristic)
└─ 부가 caveat 컬럼: "환경 제약" (예: "web_fetch: Trees API 7MB 한계 시 truncated")
```

이 형태는 다음을 모두 만족:
1. CodeQL `@precision` 메타데이터 패턴 (결정성 컬럼화).
2. Sourcegraph 2-tier 단일 축 (자동 fallback 가능).
3. Semgrep 3-tier rule confidence (enum 폭발 방지).
4. SonarCloud WARNING 부가 (환경 제약 인지).
5. Linguist byte-only 의도 (LOC 추정의 본질적 stochastic 성격 인정).
