# 아키텍처 멀티렌즈 리뷰

$ARGUMENTS (검토 대상 — 설계안, 변경 PR, 아키텍처 문서, 또는 "현재 열린 파일")를 4가지 독립 관점으로 병렬 검토한다.
합의를 찾는 게 아니라 **관점 간 모순을 드러내는 것**이 목표다.

## 언제 사용하나

- 새 플랫폼 컴포넌트 도입 결정
- 기존 아키텍처 대규모 변경 (K8s 마이그레이션, 미들웨어 교체, 네트워크 재설계)
- 장애 후 재발방지 설계 검토
- PR 머지 전 구조적 의사결정 검증

## 실행 흐름

### 1단계: 정보 신뢰도 선언

검토 대상 문서/변경안의 정보 풍부도를 선언한다.

| 등급 | 기준 |
|---|---|
| A | 전체 파일/설계 문서 + 환경 컨텍스트 확보 |
| B | 일부 파일 또는 고수준 설명만 | 
| C | 구두 설명만, 파일 미확보 |

C등급이면 검토 전 파일/문서를 먼저 요청한다.

### 2단계: 4개 에이전트 병렬 생성

**같은 메시지에 4개 Agent 호출을 동시에** 실행한다 (순차 실행 금지).

| 에이전트명 | 렌즈 | 핵심 질문 |
|-----------|------|---------|
| `reliability-lens` | 신뢰성 | "이 설계가 실패한다면 어떤 경로로?" |
| `security-lens` | 보안 | "이 설계를 공격/악용한다면 어떻게?" |
| `ops-complexity-lens` | 운영복잡도 | "이 설계가 새벽 3시 장애를 유발한다면?" |
| `cost-lens` | 비용·지속가능성 | "이 설계가 6개월 후 운영 부담이 된다면?" |

각 에이전트 prompt 템플릿:

```
너는 [{렌즈명}] 관점의 인프라 리뷰어다.

검토 대상:
{$ARGUMENTS 또는 첨부 파일 내용}

회사 환경:
- DEV: Kubernetes + ArgoCD (air-gap, Harbor DEV)
- STG/LIVE: Docker Compose (Harbor STG/LIVE, Infisical)
- HIWARE 제약: 로컬→서버 직접 SSH 불가

너의 역할: [{렌즈명}] 관점에서만 검토한다. 다른 렌즈는 신경 쓰지 않는다.

출력 형식:
1. **주요 우려사항** (3~5개, 심각도 순): 각 항목마다 "왜 이게 문제인가" + "어떤 조건에서 발생하는가" 명시
2. **반드시 해소해야 할 레드라인** (있으면): 이 조건이 충족되지 않으면 설계 진행 불가
3. **수용 가능한 트레이드오프** (있으면): 이 렌즈 관점에서 감수할 수 있는 약점

반드시 지킬 규칙:
- 근거 없는 주장 금지. 공식 문서 또는 실 사례 인용
- "best practice라서"는 근거가 아님. 이 환경에서 왜 문제인지 명시
- 불확실하면 `[추측]` 라벨
```

### 3단계: 진행 상황 안내

"4개 렌즈 에이전트 병렬 검토 중입니다. 완료 순서대로 결과가 도착합니다." 라고 안내한다.

### 4단계: 모순점 교차 분석 (4개 완료 후)

4개 리포트를 받은 후 team-lead가 교차 분석한다.

**교차 분석 구조:**

```markdown
## 렌즈 간 모순 (핵심)

| 항목 | 신뢰성 | 보안 | 운영복잡도 | 비용 |
|------|--------|------|-----------|------|
| [항목1] | 긍정/우려 | 긍정/우려 | ... | ... |

**모순 1**: [신뢰성] vs [보안] — 신뢰성은 A를 권장하는데 보안은 A의 X 위험을 지적함
  → 해소 방안: ...

**모순 2**: ...
```

모순이 없으면: "4개 렌즈 모두 동일한 우려를 공유함 → 구조적 문제 신호"로 해석한다.

### 5단계: 종합 판정

```markdown
## 종합 판정

**진행 가능 여부**: ✅ 조건부 진행 / ⚠️ 수정 후 진행 / ❌ 재설계 필요

**레드라인** (하나라도 해소 안 되면 진행 불가):
- [ ] ...

**권고 수정 사항** (진행 전):
1. ...

**수용 가능한 트레이드오프**:
- ...

**다음 단계**: 수정 후 infra-reviewer 또는 writing-plans로 구체 구현
```

## 주의사항

- 4개 Agent는 **반드시 같은 메시지에서 동시에 호출**한다. 순차 호출 시 첫 Agent 결과에 후속 Agent가 편향됨
- team-lead의 역할은 합의 도출이 아니라 **모순 식별**. 모든 렌즈가 같은 결론이면 "합의됨"이 아니라 "다른 각도를 놓쳤을 수 있음"으로 의심
- 각 Agent에게 다른 Agent의 결과를 보여주지 않는다 (독립성 유지)
