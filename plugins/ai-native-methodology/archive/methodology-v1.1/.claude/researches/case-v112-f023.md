# Case Research (F-023): Tarjan SCC + Spring Modulith / ArchUnit 실무 사례

> 작성일: 2026-04-28
> 역할: 테크기업 사례 리서처 (F-023 단독)
> Time budget: 8분 hard limit (실제 ~3분 사용)

---

## §1. Executive Summary

실무 사례 리서치 결과는 **document 입장 (자동 high 격상)** 과 **senior 입장 (BC 미정의 시 medium + decision_required)** 사이에서 **senior 쪽으로 약 60:40 기울임**. 핵심 근거:

1. Spring Modulith 창시자 Oliver Drotbohm 이 GitHub Discussion #493 에서 **"domain-legitimate cycle" 개념을 명시적으로 거부**했으나, 동시에 **interface inversion 같은 우회 기법**을 권장해 "기계적 high → 즉시 차단"이 아닌 **"기계적 탐지 → 도메인 의도 기반 리팩터링 결정"** 패턴을 권장.
2. Spring Modulith 공식 문서가 legacy 대응으로 **`@ApplicationModule(type=Type.OPEN)`** 및 **`detectViolations().filter(...)`** 우회 API 를 1급 시민으로 제공 — 즉 도구 자체가 "BC 미정의 코드베이스는 cycle 을 즉시 fail 시키지 않는다" 를 인정.
3. ArchUnit 의 **FreezingArchRule** 는 "기존 위반은 baseline 으로 수용, 신규 위반만 차단" — 이는 **F-023 의 BC 미정의 default = medium + decision_required** 와 정확히 동치인 운영 패턴.
4. 단, 어떤 공식 문서/유명 사례도 "intent" 라는 4-way enum 어휘를 쓰지 않음 → schema 어휘는 **3값 + boolean (senior 안)** 이 더 산업 표준에 가까움.

따라서 **격상은 v1.1.2 가능하나, F-023 의 default policy 는 senior 안 채택 권장**.

## §2. Spring Modulith 실무 채택 사례

### 2.1 Oliver Drotbohm (창시자) 의 실무 가이드

**1차 사료: GitHub Discussion #493 ("Circular dependencies working with events and aggregate associations")**

질문자는 OrderService → OrderCompleted event → ProductService 시나리오에서 catalog ↔ order 사이클이 탐지된 실무 케이스를 제시. Drotbohm 답변 요지:

- **"A directed acyclic graph between modules is key to an evolvable architecture"** — cycle 합법화 거부
- **휴리스틱**: "Which of the two modules could exist without the other?" — 단방향성 강제 결정 프레임
- **기술적 우회**: "introduce an interface on the catalog side for OrderCompleted to implement" — 즉, **알고리즘이 high 로 잡으면 도메인 분석 후 inversion 으로 해결**

**해석 (F-023 관점)**:
- Drotbohm 도 "BC 내부 cycle 은 OK" 를 인정하지 않음 → document 입장 지지
- 그러나 그의 해법은 **"즉시 fail" 이 아닌 "decision_required 후 inversion"** → senior 의 `decision_required` 플래그와 부합
- 즉 algorithm = ground truth 이지만 **severity 는 해석 후 결정** 이 사실상 표준

**2차 사료**: Spring blog "Introducing Spring Modulith" (2022-10-21) — `verify()` 호출이 cycle 발견 시 예외 throw 가 default 임을 명시.

### 2.2 Spring Modulith 공식 verification API 분석

**URL**: https://docs.spring.io/spring-modulith/reference/verification.html

- `verify()` default: 위반 발견 시 **즉시 예외**
- 우회 API: `detectViolations().filter(violation -> ...).throwIfPresent()`
- **차별 없음**: "cycles within a single module" vs "cycles across modules" 구분 미제공 → 모든 cycle 은 module 경계 cycle 만 평가
- **legacy 탈출구**: `@ApplicationModule(type=Type.OPEN)` 으로 module 을 "open" 선언 시 검증 완화

**해석**: 공식 문서는 "BC 내부 cycle = 정상" 을 직접 명시하지 않지만, **`detectViolations().filter()` API 자체가 "수용 가능한 위반" 의 존재를 인정**. 이는 senior 의 `decision_required` 와 의미상 동등.

### 2.3 컨퍼런스/사례

- **Oliver Drotbohm — "Refactoring to a System of Systems"** (Speaker Deck): legacy → modulith 점진적 전환 시 module 경계가 "선언적이지만 강제는 단계적" 패턴 권장.
- **Spring I/O 2019 — "Building better monoliths"**: 동일 저자, 동일 메시지.
- **JetBrains IDEA Blog (2026-02) — "Migrating to Modular Monolith"**: tutorial 성격이지만 cycle 처리는 "elimination only" 단일 노선 (events / common 이동 / responsibility 재설정). 단, **이는 "BC 가 이미 정의된" 상태 가정**.

## §3. DDD + cycle detection 실무

### 3.1 Vernon IDDD-Samples 분석

**URL**: https://github.com/VaughnVernon/IDDD_Samples

- 4개 BC 분리: `iddd_agilepm` / `iddd_collaboration` / `iddd_identityaccess` / `iddd_common`
- **각 BC 는 독립 Maven 모듈** — 즉 BC 간 cycle 은 구조적으로 발생 불가
- BC 내부 (`com.saasovation.agilepm.domain.model.product` 등) 는 aggregate 단위 패키지 분리

**핵심 관찰** (코퍼스 기반 — 직접 ArchUnit verify 미실행):
- IDDD-Samples 는 **BC 경계 = 모듈 경계 = 빌드 단위** 로 일치 → cycle detection 자체가 BC 경계와 1:1
- Vernon 의 책 (IDDD Ch.4 "Architecture") 도 "aggregate 간 참조는 ID 만 (reference by identity)" 강제 — **BC 내부 aggregate 사이클은 의도적으로 차단되는 설계**
- 즉, **"BC 내부 cycle 이 정상"** 이라는 senior 의 주장은 IDDD 정전(正典) 코드베이스에서는 관측되지 **않음**. 이는 document 입장 보강.

**(코퍼스 기반 — URL 본문 미직접검증; structure 만 검증됨)**

### 3.2 ArchUnit production 사용 사례

**URL**: https://www.archunit.org/userguide/html/000_Index.html
- 공식 user guide: **"slices should be free of cycles"** — universal rule, BC 차별 없음
- **그러나 FreezingArchRule 제공**: "first run records all current violations as baseline; subsequent runs only fail on new violations"
- **"refreezing" 메커니즘**: "store will just be updated with current state, reported result will be success" — 즉 의식적으로 cycle 을 baseline 으로 수용 가능

**URL**: https://deepwiki.com/TNG/ArchUnit/2.3.2-freezing-architecture-rules
- 명시적 use case: **"introducing a new ArchRule to an existing project that causes too many violations to solve at the current time — a typical example is a huge legacy project where a new rule might cause thousands of violations"**
- 즉 ArchUnit 팀 자체가 **"legacy = 기존 cycle 수용, 신규 cycle 만 차단"** 패턴을 1급 사용 사례로 인정

**해석 (F-023 관점)**: FreezingArchRule = senior 의 `decision_required` 의 산업 표준 구현. 알고리즘이 high 로 잡았더라도 **"baseline 합의 후 점진 해결"** 가 정상 운영 패턴.

### 3.3 jmolecules-integrations Issue #91 — false positive 사례

**URL**: https://github.com/xmolecules/jmolecules-integrations/issues/91 (search 인용, 직접 fetch 미수행)
- Title: "False positive for ArchUnit aggregate reference verification"
- 시사점: aggregate 참조 검증에서도 false positive 가 보고됨 → 알고리즘 출력 = ground truth 가 아님을 jmolecules (Drotbohm 본인 프로젝트) 에서도 인정

## §4. Anti-cases (자동 high 가 해로운 경우)

### 4.1 ArchUnit Issue #361 — Cycle Report 가독성 문제

**URL**: https://github.com/TNG/ArchUnit/issues/361 (search 인용)
- Tarjan SCC 출력이 너무 broad → 어느 cycle 이 진짜 문제인지 우선순위 매기기 곤란
- **시사점**: 알고리즘 출력의 raw severity 는 "high 일괄" 로 처리 시 노이즈 폭발 가능 → senior 의 "BC 미정의 = medium" 디폴트가 노이즈 완화책으로 합리적

### 4.2 ArchUnit Issue #298 — Final/Interface 변수 cycle 미탐지 (false negative)

**URL**: https://github.com/TNG/ArchUnit/issues/298 — 알고리즘 자체의 한계: false negative 도 존재 → 알고리즘은 신호이지 판결 아님.

### 4.3 코퍼스 기반 패턴 (URL 미검증)

레거시 자바 모놀리스 (10년+) 에 ArchUnit 처음 도입 시:
1. 1차 verify() → 수백 cycle finding
2. 팀이 압도되어 ArchUnit 자체를 disable 하거나 `@ArchIgnore` 남발
3. → **FreezingArchRule baseline 합의 → 점진 해결** 가 "건강한 도입" 으로 정착

이 패턴은 ArchUnit user guide 가 FreezingArchRule 을 1급으로 명시한 이유와 직결.

## §5. 결론 — document vs senior 충돌 해결

| 쟁점 | document 입장 | senior 입장 | 실무 사례 결과 |
|---|---|---|---|
| 격상 범위 | 4건 일괄 v1.1.2 OK | F-023 은 v1.2.0 MINOR | **문서 쪽 유리**: 도구/창시자 모두 cycle 을 architectural concern 으로 명확히 다룸 → 격상 자체는 v1.1.2 PATCH 범주 가능. 단 default 정책만 senior 안 채택 시 MINOR 불필요 |
| BC 미정의 default | low + decision_required | medium + decision_required | **senior 우세**: ArchUnit FreezingArchRule, Spring Modulith `Type.OPEN` 둘 다 "기존 cycle = baseline 수용, 즉시 fail 안 함" 산업 표준. 단 severity 는 low 가 아니라 **medium** 이 정합 (algorithm 신호는 무시 못 함) |
| schema "intent" 단어 | 4-way enum | 3값 + boolean 페어 | **senior 우세**: Spring Modulith / ArchUnit / Vernon 모두 "intent" 어휘 미사용. 산업 표준은 binary "violation y/n" + boolean "frozen/accepted" 패어 → senior 의 3값 + boolean 이 더 정합 |
| 알고리즘 vs 도메인 우선 | hybrid (탐지 + BC 분기) | hybrid (동일) | **만장일치**: Drotbohm 답변 ("DAG is key... but use interface inversion") 은 정확히 "알고리즘 탐지 → 도메인 의도 기반 결정" hybrid. 두 입장 일치 — 충돌 없음 |

### 최종 권고 (실무 데이터 기반)

- **격상 자체**: v1.1.2 PATCH 로 충분 (document 입장 채택, MINOR 불필요)
- **BC 미정의 default**: **`medium + decision_required`** (senior 안 채택) — FreezingArchRule / Type.OPEN 산업 표준과 일치
- **schema 어휘**: **3-state severity (low/medium/high) + boolean `frozen` 또는 `accepted_baseline`** (senior 안 변형) — 4-way "intent" enum 은 산업 표준 어휘에 없음
- **algorithm vs domain**: hybrid 그대로 유지 (양 입장 일치)
