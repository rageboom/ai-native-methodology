# Case Research (F-007): inventory schema 신설 vs 보류 — 실무 사례

> 작성일: 2026-04-28
> 리서처: 테크기업 사례 리서처 (Case Study Researcher)
> 시간 사용: 약 5분 / 8분 한도, WebFetch 6회 / WebSearch 3회

---

## §1. Executive Summary

산업 사례를 종합하면 **document agent 손을 들어주는 비중이 더 크다**. 단, 조건부로.

1. **Backstage, CycloneDX, Cookpad, OpenAPI 3.1** 등 주요 표준/플랫폼은 모두 **JSON Schema를 SoT로 게시**한다. `Entity.schema.json` 같은 정식 스키마 파일이 GitHub에 공개되어 있고, 별도 example/문서는 자동 생성되거나 schema에 종속된다.
2. 그러나 **CI 자동 검증이 함께 도입되지 않은 schema는 실제로 dead weight화 사례가 존재**한다 — OpenAPI 2/3과 JSON Schema 간 7년 이상 divergence가 지속되며 ecosystem 비용이 누적된 것이 대표적.
3. **결론**: schema 신설 자체는 산업 표준이나, "신설 + CI 검증 동시 도입" 또는 "신설 + §4.2 example 자동 생성" 둘 중 하나여야 senior agent가 우려한 SoT 분산을 회피할 수 있다.
4. v1.1.2에서 schema만 신설하고 CI를 v1.3.0까지 미루는 것은 **OpenAPI가 7년간 겪은 실패 패턴을 그대로 재현할 위험**이 있다.

## §2. Catalog/Inventory schema 산업 사례

### 2.1 Backstage Catalog Info (Spotify)

- **포맷**: catalog-info.yaml (YAML, JSON 호환)
- **JSON Schema 게시 여부**: **YES**. `packages/catalog-model/src/schema/Entity.schema.json`, `EntityMeta.schema.json` 등이 GitHub에 공개되어 있음.
- **엔티티 종류 (9종)**: Component, API, Resource, System, Domain, Group, User, Template, Location
- **검증 시점**: 공식 문서상 **ingestion time only** (CI-level 검증은 별도 도구 필요)
- **CI 도구 생태계**:
  - RoadieHQ `backstage-entity-validator` (CLI/STDIN)
  - GitHub Action `backstage-entity-validator` (Marketplace 등재)
  - Backstage 내장 `entityKindSchemaValidator()` 함수
- **시사점**: schema는 신설하되 **CI 검증은 third-party 도구로 위임**한 모델. AI-Native 방법론이 v1.1.2에서 schema만 게시하고 CI를 미루는 시나리오와 정확히 일치하나, Backstage는 검증 도구 생태계가 빠르게 따라붙음 (schema 게시 후 1년 이내 GitHub Action 등장).

### 2.2 CycloneDX SBOM (OWASP)

- **JSON Schema 게시 여부**: **YES**. 정식 versioned JSON Schema가 표준 커뮤니티에서 게시.
- **CI 검증 채택률**: **매우 높음**. 주요 도구:
  - `CycloneDX/sbom-utility` — CI tool chain 통합 명시
  - `cdxgen` — CI/CD 파이프라인 자동 제출 (Dependency Track)
  - GitHub Marketplace에 Go, .NET, Node.js, Erlang/Elixir용 CycloneDX SBOM Action 다수
- **schema-as-SoT 정도**: tools.json도 schema 검증 통과 필수 (2025-03 시점). 즉 **tooling 자체가 schema에 종속**.
- **시사점**: CycloneDX는 **schema 신설 + CI 검증 + 도구 ecosystem이 동시에 굴러가는 모델**. document agent가 권장한 "schema 신설"의 이상적 형태.

### 2.3 SLSA Provenance Schema

- **JSON Schema 게시 여부**: **NO** (정확히는 JSON Schema 형식 아님). protobuf schema (`provenance.proto`) + 인라인 JSON 구조 문서로 유지.
- **이유 추정**: snake_case (proto) ↔ lowerCamelCase (JSON serialization) 변환 규칙이 복잡하여 단일 JSON Schema로 표현 시 정보 손실.
- **CI 검증**: 명세 자체에서 다루지 않음. 구현 도구가 자체 검증.
- **시사점**: **공식 표준이라 해서 반드시 JSON Schema를 발행하는 건 아니다**. SLSA처럼 "확장 가능 + consumer가 unrecognized field 무시" 정책이면 schema strict 검증보다 **convention 문서**가 더 적합. 이는 senior agent 의견에 부분적 근거 제공.

## §3. Schema와 example의 SoT 분산 위험 — 실증 사례

### 3.1 OpenAPI 3.0→3.1 divergence 사례 (확정)

- **divergence 기간**: 약 **7년 (2017~2024)**. OpenAPI v2는 JSON Schema draft 4 기반이지만 deviation 다수, v3은 draft 5 기반인데 JSON Schema는 draft 7로 진행.
- **구체적 충돌 예**: OpenAPI는 `type: string` + `nullable: true`, JSON Schema는 `type: [string, null]`. → ReDoc은 `[string, null]` 허용, Speccy linter는 invalid OpenAPI라며 거부. 즉 **같은 의도를 가진 데이터를 표현하는 정답이 도구별로 다름**.
- **비용**: 도구 vendor의 dual maintenance, end-user의 portability 손실.
- **해결**: **OpenAPI 3.1 (2021-02)에서 JSON Schema Draft 2020-12 full alignment**. 2026 현재 3.1 채택률이 tipping point 통과, 3.0 유지 팀은 호환성 벽에 부딪힘.
- **시사점**: senior agent의 "5년 후 divergence 불가피" 우려는 **실증된 패턴**. 단, OpenAPI가 결국 alignment로 수렴한 점은 SoT 분산이 영구적이진 않다는 반증.

### 3.2 AsyncAPI 1.x→2.x 사례

- **확보 데이터**: 부족. WebSearch 결과 직접 hit 없음. (코퍼스 기반 — URL 미검증) AsyncAPI는 OpenAPI 패턴을 따라 2.x에서 schema alignment를 강화한 것으로 알려져 있음.

### 3.3 schema 신설 후 dead weight 화 사례

- **JSON Schema 자체의 메타-사례**: GitHub Discussion #557 (json-schema-org)에서 "stable spec" 제안 — **schema 버전 난립이 사용자/구현자/유지보수자 모두에게 부담**이라는 인정.
- **schema drift 실제 사고**:
  - 핀테크: balance 필드가 number → string으로 변경되어 reconciliation service가 합산 대신 문자열 concatenation 수행. 탐지에 4일, 정리에 2주 소요.
  - 데이터 파이프라인: 외부 vendor가 JSON payload 중간에 컬럼 추가했으나 schema 검증 미작동 → 침묵 실패.
- **시사점**: schema 자체보다 **schema validation이 실행되지 않을 때**가 진짜 dead weight. senior agent 지적 정확.

## §4. Schema 가치의 전제 — CI 자동 검증 채택률

### 4.1 산업 표준: schema validation in CI 채택률

- **CycloneDX**: GitHub Marketplace 6개 이상 언어별 Action. **사실상 default**.
- **Backstage**: ingestion time 검증 + third-party CI Action 1개 (Marketplace). **option but common**.
- **JSON Schema 자체 (Cookpad 사례)**: schema가 frontend form generator + backend validator를 동시에 구동. **실행 안 되는 schema 없음**.
- **OpenAPI**: 3.1로 alignment된 후 schema-driven contract testing이 표준 워크플로우 (oasdiff 등 PR review 자동화).

→ **현대 산업 표준은 schema = CI 검증 동반**. CI 없는 schema 게시는 점점 minority.

### 4.2 IDE autocomplete (JSON Schema → IntelliSense) 활용도

- **Backstage**: `$schema: "https://backstage.io/schema/config-v1"` 선언으로 IDE autocompletion 지원.
- **VS Code**: `json.schemas` setting + `$schema` 키워드로 완전 자동.
- **활용도**: schema 게시 시 **추가 비용 없이 얻는 부수 효과**. document agent가 언급한 "IDE autocomplete" 가치는 검증된 사실.
- **단**: 이 가치는 schema가 **정확하고 최신**일 때만 작동. 5년 묵힌 schema는 IDE에서 잘못된 autocomplete를 제공해 오히려 해악.

## §5. 결론 — document vs senior 충돌 해결

| 쟁점 | document (신설 OK) | senior (보류 권장) | 실무 사례 |
|---|---|---|---|
| schema 신설 자체의 정당성 | JSON Schema 표준 + AsyncAPI 패턴 | CI 미도입 시 dead weight | **document 우세**. Backstage, CycloneDX, Cookpad 모두 schema 게시. SLSA만 예외 (proto 사용) |
| SoT 분산 위험 | 호환 설계로 회피 가능 | 5년 후 divergence 불가피 | **senior 입증**. OpenAPI/JSON Schema 7년 divergence (2017~2024)는 실증. 단, 결국 3.1에서 수렴 |
| schema dead weight 위험 | 검증 자동화로 방어 | CI 없으면 무의미 | **senior 우세**. fintech/data pipeline drift 사고 사례 다수. CycloneDX는 CI 도구가 schema와 동시 출시되어 회피 |
| IDE autocomplete 가치 | 추가 비용 없이 획득 | 미언급 | **document 우세**. Backstage `$schema` 선언으로 즉시 작동 |
| 신설 시점 | v1.1.2 | CI 도입 시점 (예상 v1.3.0) | **타협안**: v1.1.2에 schema + minimal CI (예: pre-commit ajv 검증) 동시 도입 권장 |

### 최종 권고 (실무 사례 기반)

**document와 senior 모두 부분적으로 옳다**. 산업 사례는 다음을 시사:

1. **Schema 신설은 do (산업 표준)** — Backstage/CycloneDX/Cookpad 모델 따라가기.
2. **단, "schema only, CI later" 패턴은 OpenAPI 7년 divergence를 재현할 위험** — senior agent 우려 정확.
3. **권장 패치 시나리오**:
   - **(A) schema + §4.2 폐지 (schema = SoT, §4.2 자동 생성)** — senior agent 대안, Cookpad 모델, 가장 깔끔.
   - **(B) schema + 최소 CI (pre-commit `ajv validate` 또는 GitHub Action)** — Backstage 모델, v1.1.2에서 즉시 가능.
   - **(C) schema 보류, §4.2 example만 유지** — senior agent 1순위, SLSA 모델. AI-Native 방법론처럼 LLM이 매번 inventory.json을 생성하는 컨텍스트에서는 example만으로도 충분할 수 있음.
4. **PoC #01에서 inventory.json을 즉석 발명한 점**은 schema 부재 비용을 이미 입증 → **(A) 또는 (B)가 (C)보다 우세**해 보임.
