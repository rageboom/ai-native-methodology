# Smilegate UX/UI팀 위키 기획서 작성 스킬

Smilegate UX/UI팀의 기획서를 Confluence 위키 페이지로 작성하는 스킬.
기획서 본문은 위키 텍스트/표로 작성하고, Figma 화면 설계는 링크로만 참조한다.

> **왜 위키인가?**
> AI(Claude)가 Confluence MCP로 위키 페이지를 직접 읽고 쓸 수 있다.
> Figma 임베드(iframe)나 이미지는 AI가 내용을 읽을 수 없으므로 사용하지 않는다.

## 시작 전 — 세 가지 분기 판단

요청을 받으면 반드시 세 가지를 먼저 판단하고, 모호하면 한 번에 묶어서 확인한다.

### 분기 1: 신규 vs 운영 기획

| 신호 | 판단 |
|------|------|
| "새로 만들어줘", "처음부터 기획", "OOO 시스템 기획해줘" | **신규 기획** — 문서 이력 1행, 전체 섹션 구조 설계 |
| "변경", "추가", "개선", "보완", "이미 있는데 ~ 부분만" | **운영 기획** — 문서 이력 누적, 변경 섹션만 집중 |

### 분기 2: 공유 대상

| 신호 | 판단 |
|------|------|
| "팀 내 공유", "내부 리뷰" | **맥락·방향 중심** — 의도, 미결 사항 포함 가능 |
| "개발팀 전달", "개발팀 넘길" | **명세 중심** — 컴포넌트·상태·예외·API 연동 빠짐없이 |

### 분기 3: Figma 작업 포함 여부

| 신호 | 판단 |
|------|------|
| "Figma도 작업해줘", "피그마 프레임도" | 위키 작성 완료 후 Figma MCP로 별도 진행 |
| Figma 언급 없음 | 위키 기획서만 작성. 화면 설계 참조(6절)는 링크 자리 비워둠 |

## 워크플로우 — 위키 기획서 작성

### 1단계: 템플릿 읽기 (필수)

작업 시작 전 반드시 `references/template.md`를 `view`로 읽는다.
이 파일이 위키 페이지에 입력할 섹션 구조와 표 형식의 기준이다.

### 2단계: 기획서 내용 초안 작성

템플릿 구조에 맞게 내용을 채운다.

| 섹션 | 내용 | 팀 내 | 개발팀 |
|------|------|:-----:|:------:|
| 1. 배경 | 목적·범위·관련 문서 | ✓ | ✓ |
| 2. 정의 | 핵심 용어 정의 | ✓ | ✓ |
| 3. 주요 기능 | 시스템 핵심 기능 정의 | ✓ | ✓ |
| 4. 프로세스 | 단계별 흐름·분기 조건 | ✓ | ✓ |
| 5. 화면 구성 | 메뉴 구조(LNB)·화면 목록 | ✓ | ✓ |
| 6. 화면 설계 참조 | Figma 링크만 | 링크 | 링크 |
| 7. 사용 정책 | 등록·조회·수정·삭제 | 요약 | 상세 |
| 8. 상태/처리 정책 | 상태값·전이 규칙 | ✓ | ✓ |
| 9. 예외 처리 | 비정상 상황별 처리 | 주요만 | 전체 |
| 10. 확인 필요 사항 | 미결 항목 관리 | ✓ | ✓ |

**개발팀 전달용 추가 필수 항목:**
- 에러 상태·로딩 처리·빈값 케이스 명시 (9절)
- API 연동 포인트 명시 (4절 또는 별도)
- 컴포넌트 단위 상태 정의 (7절)

### 3단계: Confluence MCP로 위키 페이지 생성

```
tool_search("confluence page create")  →  Atlassian:createPage 또는 wiki_page_create
```

페이지 생성 시 필수 파라미터:
- `title`: 시스템명 + " 기획서" (예: "차량관리 시스템 기획서")
- `spaceKey` 또는 `space`: 사용자에게 확인
- `parentId` 또는 `parentPageTitle`: 사용자에게 확인
- `content`: 2단계에서 작성한 내용 (Confluence storage format 또는 wiki markup)

> **주의**: Figma 임베드 매크로(`<ac:structured-macro ac:name="figma">`)는 사용하지 않는다.
> AI가 이후 페이지를 읽을 때 iframe 내용을 추출할 수 없기 때문이다.
> Figma 링크는 일반 하이퍼링크(`[링크 텍스트](URL)`)로만 삽입한다.

### 4단계: 생성 확인 및 보고

페이지 생성 완료 후:
1. 생성된 위키 페이지 URL을 사용자에게 전달한다.
2. 미확정 항목(`[미확정]`)이 있으면 10절에 Q-번호로 등록했음을 안내한다.
3. Figma 작업이 필요한 경우 다음 단계를 안내한다.

## (선택) Figma 프로세스 다이어그램 작업

위키 기획서 작성 완료 후, 사용자가 요청하는 경우에만 진행한다.

```
tool_search("figma diagram")  →  Figma:get_figjam 또는 Figma:get_design_context
```

- 4절 프로세스 흐름을 기반으로 FigJam 다이어그램 생성
- 생성된 Figma 링크를 위키 페이지 6절 표에 업데이트
- 도형은 12개 이하로 제한
- 분기 조건은 4절의 "→ Y / → N" 구조 그대로 반영

## 플레이스홀더 처리 규칙

- `{{변수명}}` 형식은 모두 실제 값으로 교체
- 정보가 없으면 임의로 채우지 말고 사용자에게 확인
- `[미확정]` 항목은 반드시 10절 확인 필요 사항에 Q-번호로 등록

---

## 작업 원칙

1. 위키 페이지 내용은 텍스트와 표로만 작성한다 (이미지·임베드 금지)
2. Figma는 링크로만 참조한다
3. 정보 부족 시 임의 추정하지 않고 사용자에게 확인한다
4. 위키 페이지 생성 완료 → URL 보고 → 사용자 검토 대기 → 다음 진행
5. 운영 기획 수정 시 기존 페이지의 문서 이력을 반드시 누적한다

---

## (선택) 작성 완료 후 — 설계 빈틈 점검 안내

기획서 작성을 마친 뒤, 필요한 경우 프로세스 설계의 빈틈(책임 누락·예외 미대응·흐름 단절 등)을 점검할 수 있다.
점검을 원하면 **QA 플러그인의 `qa-process-eval` 스킬**을 참조한다.
